# First-cut results: four libraries vs alien-signals v3.2.1

2026-07-03, final verification pass. Independent re-runs of everything; no
numbers below are taken from the implementation agents' reports.

## Methodology

- **Machine**: macOS arm64, Node v24.16.0, ambient load average 3.5–8.6
  during runs (recorded per result file in JSON metadata). Not an idle
  machine; see caveats.
- **Conformance**: `FRAMEWORK=<fw> pnpm -C harness conformance`
  (reactive-framework-test-suite v0.0.2 via vitest 4.1.6, 179 cases), one
  fresh run per framework at 20:17–20:18.
- **Bench**: `pnpm -C harness bench --frameworks alien-v3,control,sweep,arrayd,arena`
  (all four suites; kairo includes molBench). One esbuild-bundled child
  process per (framework, suite) pair under `node --expose-gc`; the milomg
  harness reports **fastest-of-N** per case. The full matrix was run
  **3 times, interleaved by framework** (each pass runs alien-v3 → control
  → sweep → arrayd → arena before the next pass starts). Cells below are
  **median-of-3** of the fastest-of-N values; spread = (max−min)/median
  across the 3 passes.
- **Memory**: `pnpm -C harness memory` for all five, single run per
  framework in its own `--expose-gc` child; retained KB **through the
  shared adapter** (uniform `{read,write}` wrapper overhead — compare
  relatively, not against upstream's raw-API numbers).
- **Correctness at bench scale**: all 60 bench children (3 passes × 5
  frameworks × 4 suites) exited 0 with **zero `console.assert` failures**
  (cellx asserts exact values; dynamic ran with `testPullCounts: true`,
  asserting exact computed-evaluation counts).
- **Raw outputs** (all in `harness/results/`):
  - pass 1: `2026-07-03T20-18-28-<fw>.{json,csv}` + `bench-pass1.log`
  - pass 2: `2026-07-03T20-30-11-<fw>.{json,csv}` + `bench-pass2.log`
  - pass 3: `2026-07-03T20-41-26-<fw>.{json,csv}` + `bench-pass3.log`
  - memory: `2026-07-03T20-52-34-memory-<fw>.{json,csv}` + `memory-run.log`

### Caveats

- Ambient load was nonzero throughout (loadavg spiked to ~8 during part of
  pass 3). Median-of-3 of fastest-of-N is robust to one-sided interference,
  and per-cell spreads bear that out: **all spreads ≤ 16% except one** —
  alien-v3 `createSignals` hit 15.68 ms in pass 2 (vs 3.45/3.93 in passes
  1/3; 311% spread). For that one cell trust the min (3.45, matching the
  18:00 baseline's 3.33).
- Memory is a single run per framework (the probe is deterministic modulo
  GC timing; numbers match the implementation agents' runs to within ~1%).
- Fastest-of-N systematically hides GC cost, which flatters
  allocation-heavy designs and understates arena/arrayd's GC-pressure wins.
  Mean/p99 + GC-time reporting is still TODO (RESEARCH.md §1.8).
- A sixth adapter (`arena-links`, the arena de-risk variant) exists in the
  registry but was not part of this verification scope.

## Analysis: predictions (IDEAS.md) vs outcomes

### control — "tuned alien-v3 … raises the bar every library must clear"

**Prediction broke, informatively.** Control is exact parity with alien-v3
in this session: 18 of 20 timed rows within ±4%, and both of its "wins"
(updateSignals −1%, 6-100x15 −4%) are inside per-pass spread. The two
optimizations from the brief measure as zero:

- **Persistent stacks**: the cons cells they eliminate are young-gen
  objects that die instantly — V8's scavenger already makes them nearly
  free (RESEARCH.md §6 GC economics). No suite shape allocates enough
  propagation frames for it to matter.
- **Quiet epoch**: the control agent's stronger result stands up to my
  numbers — in alien's architecture the sound version is *structurally
  dead code* (every propagate is immediately preceded by an epoch bump, so
  the Pending flag always already encodes what the epoch would say;
  instrumented fast-path hit count: 0). Preact/Vue/Angular profit from a
  global version only because they have pull-verified nodes that receive
  no invalidation pushes. This kills RESEARCH.md §8.6 ("cheapest known
  win") for full-push cores.

**New finding this session**: a real regression the control smoke bench
(kairo+sbench only) never saw — `dynamic/3-5x500` (500-layer-deep graph)
is **+18% slower, consistently** (103.6/104.7/105.0 vs alien
91.1/86.4/88.3; ~1% spread on both sides). Something in the
persistent-stack checkDirty costs on very deep verification descents.
Needs root-causing before any upstreaming claim.

### sweep — "should win small graphs & write-heavy; supposed to lose big somewhere"

**The prediction held with unusual precision, and sweep beat alien in more
places than predicted.** Sweep is the fastest framework overall on kairo
(394 vs alien's 417 total), wins broad (−15%), deep (−12%), diamond (−9%),
triangle (−12%), unstable (−30%), cellx at both scales (2.3–2.6×),
createSignals (5×), and two dynamic shapes. When a write burst affects
most live effects, alien's entire push machinery is pure overhead and
sweep's tiny nodes win.

The collapse is equally clean, and the dynamic suite (which the sweep
agent never ran) quantifies it beyond updateSignals:

- `updateSignals`: **174× slower** (53.0 s vs 0.30 s). Two stacked causes:
  (a) the design's O(live effects) validation per unbatched write, and
  (b) a harness artifact that amplifies it — the bench child's
  `withBuild` wrapper keeps only the *last* scope disposer, so sbench's
  3-warmup loop leaks ~2/3 of warmup effects into the measured phase
  (~15k live effects × 400k writes; verified in `harness/bench/child.ts`
  — `scope` is overwritten per `withBuild` call). Every framework carries
  the same leak; only a global-scan design pays for it. The sweep agent's
  clean-registry repro puts the intrinsic (leak-free) cost of the same
  update patterns at ~139 ms for update1to1000 — still a loss, but ~10²
  smaller. Both numbers are true; the workload property that matters is
  *live-but-unaffected effects exist*, which is realistic.
- `4-1000x12-dyn5%`: **6.8× slower**; `25-1000x5`: **2.6× slower** — wide
  graphs where each batched write affects a small fraction of the ~1000
  live leaf effects, so the per-flush full-registry validation is almost
  all waste. `2-10x5-lazy80%`: 1.38× — 600k tiny batches, each paying the
  fixed flush/validation constant on a 50-node graph.

**The crossover curve the brief asked for, measured**: sweep wins when
(affected effects / live effects) per flush is high (cellx ≈ all, kairo
broad/unstable) or the graph is small and read-heavy; it loses roughly
linearly in the number of live-but-unaffected effects per write
(1.4× at ~10 idle effects × 600k writes, 2.6–6.8× at ~1000, 174× at ~15k).
Bookkeeping-free reactivity is real and fast below a few hundred live
effects — exactly the calibration baseline the portfolio wanted.

### arrayd — "the highest-confidence challenger: should win wide fan-out, creation, GC, unstable"

**Half held, half broke.**

- *Unstable* held: 0.82× (the dynamic-dep-churn win, as designed — cursor
  reconciliation without Link splices). Also mux 0.97 and molBench par.
- *Deep-chain risk* did **not** materialize: deepPropagation 0.99 (the
  stack-free degree-1 descent closed the predicted gap).
- *Wide fan-out* did **not** win: broadPropagation 1.04, triangle 1.13 —
  anod's −37% evidence did not transfer once full alien semantics
  (re-run trimming, notification-order preservation) were carried.
- *Creation* broke outright: createSignals 2.0×, createComputations 2.1×
  **slower**. The single 14-field node shape (~136 B) that keeps every hot
  IC monomorphic costs 2× the allocation of alien's 5-field signal; the
  agent's profiling showed GC at 34% of the creation workload. The
  monomorphism dividend shows up in the propagation rows (mux, unstable,
  updateSignals 0.99) but creation pays for it.
- *GC pressure* held where it counts: effects-10k memory −24%, and one
  scavenge per broad-propagation workload vs alien's 206 (agent-measured,
  consistent with my fastest-of-N numbers hiding it).

Net: arrayd is the only challenger at overall parity with alien on
propagation (kairo 427.6 vs 416.6, +2.6%, inside cross-pass noise — the
implementation agent's session had it slightly *ahead*; the ranking flips
run to run) while being strictly better on effect-heavy memory. Creation
is the single blocker.

### arena — "flagship: creation 6×, memory ~2.5×, write throughput 1.6×; risk = suite-scale lifecycle"

**The risk clause was the accurate prediction.** The spike's shape-level
DoD wins did not transfer to suite scale, confirming RESEARCH.md §7.3 for
the second time, now with full conformance:

- *Creation* held directionally but shrank: createComputations **0.64×**
  (best in field), createSignals 0.73× — not 6×. Reclamation, growth
  watermarks, boundary sweeps, and closure handles eat most of the spike's
  margin.
- *Memory* held partially: effects-10k **0.62×**, grid **0.78×** (best in
  field); signals +7% (packed value columns + handle closures).
- *Write throughput* broke: updateSignals 1.11× **slower**.
- *Propagation* is the loss column: every kairo row 1.14–1.83× slower
  (repeatedObservers 1.68, unstable 1.83). The Int32Array record walks
  plus per-op boundary checks (`maybeBoundary`, engine-swap indirection,
  E-dispatch) cost more than alien's pointer-chasing saves at these graph
  sizes. Notably `25-1000x5` (the largest, widest dynamic shape) is
  arena's one timed median-win vs alien (0.99; won 2 of 3 passes — par,
  not a clear win) — pointing the same
  direction as samara's "wins appear at 1000×1000" scaling data: this
  suite may simply be too small for the layout to pay.

### Portfolio-level conclusions

1. **Nobody beat alien-v3 across the board — and control shows why**: the
   reference implementation leaves almost nothing on the table for its own
   architecture. The two "known cheap wins" measure as zero (one provably
   dead). Beating alien requires changing what work exists, not doing the
   same work faster.
2. **Where alien does lose, it loses to *less* machinery, not more**:
   sweep (no graph at all) is the only thing that beats it broadly on
   propagation shapes, by 9–30%, and wins cellx by 2.3–2.6×. The
   featurelessness dividend is real. This strengthens the case for the
   unbuilt **#3 `pull`** (liveness split: sweep-like economics for the
   lazy region, push only where effects watch) — it directly attacks
   sweep's failure mode (O(live effects) flush) while keeping its wins.
3. **The suite-scale caveat (§7.3) is now confirmed law**: two independent
   layout designs (arrayd, arena) with green conformance each won their
   predicted micro-dimensions (creation OR memory OR churn) and gave the
   propagation wins back at suite scale. Layout alone does not beat a
   tuned object graph at these graph sizes on this VM.
4. **Memory/GC is where the layout bets actually pay today**: arena
   −38% effects / −22% grid; arrayd −24% effects with 200× fewer
   scavenges on broad propagation. For a product decision (Notion-scale
   editor graphs), those are the numbers that matter most alongside
   parity propagation — arrayd is closest to that combination now.
5. **Harness debt discovered**: (a) the `withBuild` scope-leak in
   `harness/bench/child.ts` distorts updateSignals for any global-scan
   design (and mildly pollutes every framework's update phase with dead
   effects) — fix or document next round; (b) fastest-of-N still hides
   GC — add mean/p99 + GC time; (c) the scope create/dispose benchmark
   (IDEAS cross-cutting list) remains unbuilt and is exactly where arena
   should shine.

### Next steps (in expected-value order)

1. Root-cause control's `3-5x500` +18% (deep-graph persistent-stack
   checkDirty) — it's the only real signal from the control experiment
   and gates any upstream PR.
2. Build **#3 `pull`** — both this session's headline results (sweep's
   wins and sweep's collapse) are the two halves of its design brief.
3. arrayd: attack creation (slimmer node or split signal shape with a
   monomorphism-preserving trick); it is one fix away from strict-
   improvement-over-alien territory (par propagation, better memory).
4. arena: benchmark `arena-links` (links-only arena under object nodes,
   already registered but unverified) to isolate how much of the
   propagation loss is record-walk vs dispatch/boundary overhead; add the
   scope-dispose benchmark where bulk-free should finally show.
5. Harness: fix the withBuild leak, add mean/p99+GC columns and a
   `--jitless` pass.
