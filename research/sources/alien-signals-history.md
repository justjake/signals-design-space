## 1. What alien-signals is, and its design constraints

[stackblitz/alien-signals](https://github.com/stackblitz/alien-signals) is Johnson Chu's (johnsoncodehk) research project exploring a **push-pull signal algorithm**. Per the [README](https://github.com/stackblitz/alien-signals/blob/master/README.md), it synthesizes four lineages:

- Propagation algorithm of Vue 3 (Johnson wrote Vue's reactivity optimizations too)
- Preact's double-linked-list approach ([Signal Boosting blog](https://preactjs.com/blog/signal-boosting/))
- Svelte's inner-effect scheduling
- Reactively's graph coloring ([milomg.dev/2022-12-01/reactivity](https://milomg.dev/2022-12-01/reactivity))

**Hard design constraints** (stated in README, applied to the algorithmic core [`src/system.ts`](https://github.com/stackblitz/alien-signals/blob/master/src/system.ts)):
1. **No Array/Set/Map** in the core — all graph edges are doubly-linked `Link` objects.
2. **No function recursion** — `propagate` and `checkDirty` use iterative traversal with an explicit linked stack (`Stack<T> = { value, prev }`).
3. Johnson's stated finding: *"under these conditions, maintaining algorithmic simplicity offers more significant improvements than complex scheduling strategies."*

Background: Johnson spent a long time [optimizing Vue 3.4's reactivity (vuejs/core#5912)](https://github.com/vuejs/core/pull/5912) — dirty-level tracking + lazy computed evaluation, 2.5–4.5x faster scheduling. When Vue 3.5 [switched to a Preact-like pull-based algorithm (vuejs/core#10397)](https://github.com/vuejs/core/pull/10397), he continued push-pull research in alien-signals. The algorithm is production-tested in [vuejs/language-tools](https://github.com/vuejs/language-tools) for incremental AST parsing / virtual code generation.

## 2. Current core data structures and algorithm (v3.2.1)

Total core: **262 lines** (`src/system.ts`) + 441-line default surface API (`src/index.ts`). Local paths: `/Users/jitl/src/alien-signals-opt/upstream-alien-signals/src/system.ts`, `.../src/index.ts`.

### Data layout
```ts
interface ReactiveNode { deps?: Link; depsTail?: Link; subs?: Link; subsTail?: Link; flags: ReactiveFlags; }
interface Link { version: number; dep: ReactiveNode; sub: ReactiveNode;
                 prevSub?: Link; nextSub?: Link; prevDep?: Link; nextDep?: Link; }
enum ReactiveFlags { None=0, Mutable=1, Watching=2, RecursedCheck=4, Recursed=8, Dirty=16, Pending=32 }
```
Every dependency edge is one `Link` participating in **two doubly-linked lists simultaneously**: the sub's dep list (prevDep/nextDep) and the dep's sub list (prevSub/nextSub) — 7 fields per edge (like Preact's "quad-linked" Node). One merged `ReactiveNode` type serves signals, computeds, effects, and scopes (v2 merged the earlier separate `Dependency`/`Subscriber` interfaces).

### Key mechanisms
- **`link(dep, sub, version)` — O(1) amortized dep re-tracking on stable graphs.** Before re-running, a sub sets `depsTail = undefined` (keeping `deps`); each read either (a) matches `depsTail.dep === dep` → skip, (b) matches `depsTail.nextDep.dep === dep` → stamp version, advance tail, **zero allocation**, or (c) allocates a new Link spliced in place. After the run, `purgeDeps` unlinks everything after `depsTail`. Johnson confirmed in [issue #108](https://github.com/stackblitz/alien-signals/issues/108): *"For a stable graph where deps are read in the same order every time, case 2 hits 100% — zero allocation."* Earlier versions experimented with looking further ahead in the dep list for dynamic patterns; **rejected to keep the algorithm simple** since stable graphs dominate real apps.
- **`Link.version` is a per-cycle dedup stamp, not a dirty-check version.** A global `cycle` counter increments per computed/effect re-run; `link` skips if `dep.subsTail` already carries the current version for this sub. Introduced in [commit 54fe1b3](https://github.com/stackblitz/alien-signals/commit/54fe1b3947fac5c0aecb73b0b0eaff000806c454) to fix [issue #81](https://github.com/stackblitz/alien-signals/issues/81) (reading the same signal 100x in one effect created 100 links; earlier full dedup had been removed because it caused regressions, see [vuejs/core#13620](https://github.com/vuejs/core/issues/13620)). Note it only dedups *consecutive* repeat reads (checks `subsTail` only).
- **`propagate(link, innerWrite)` — push phase.** Iterative DFS over sub-lists, allocating `{value, prev}` stack frames only at branching points (nodes with >1 subscriber). Marks subs `Pending` (not `Dirty` — that's the v2 change), calls `notify` on `Watching` nodes, recurses into `Mutable` (computed) nodes' subs. A 5-way flag-dispatch head handles recursion guards (`RecursedCheck` = currently tracking; `Recursed` = bookmark for inner writes). The README documents equivalent recursive reference versions of `propagate`/`checkDirty` for ports.
- **`checkDirty(link, sub)` — pull phase.** Iterative upward validation: if a dep is `Mutable|Dirty` → `update(dep)`; if `Mutable|Pending` → push onto stack and descend into its deps. When an update actually changes a value, `shallowPropagate` upgrades sibling subscribers from `Pending`→`Dirty` (this is the glitch-free "only recompute if a source truly changed" equality-cut, equivalent to Reactively's coloring). Returns early on first confirmed dirty.
- **Effect queue: a flat array with in-place segment reversal.** `notify` walks `effect.subs.sub` up the ancestor-effect chain, appends the chain to `queued[]`, then reverses the inserted segment so **outer effects run before inner** ones (Svelte-style ordering; fixed for equal ordering of inner effects in [#87](https://github.com/stackblitz/alien-signals/pull/87)). `flush` drains with `notifyIndex`, writing `undefined` to release references.
- **Signal write path:** signals store `pendingValue` + `currentValue` (v2's "deferred signal evaluation"): a write only sets `Dirty` on the signal itself and propagates `Pending`; the value is committed on next read (`updateSignal` compares old/new), so writing back the original value costs no recomputation.
- **Surface-level micro-tricks** (in `index.ts`): a `HasChildEffect = 64` flag bit outside `ReactiveFlags` gates the dispose-children slow path so leaf effects skip a deps walk; node kind discrimination via `'getter' in node` / `'currentValue' in node` (a JS-specific idiom the Dart port had to replace — [PR #90 area, see f1391ce]); `isSignal` checks `fn.name === 'bound ' + signalOper.name` on the bound closure; `batchDepth`/`startBatch`/`endBatch` are kept instead of `batch(fn)` because the closure allocation was measured at ~300x more memory over 10k iterations ([issue #96](https://github.com/stackblitz/alien-signals/issues/96): 3.23 KB vs 948.8 KB, 0.41 ms vs 0.53 ms).

## 3. Version history and why things changed

Only 3 GitHub releases have notes: [v1.0.0](https://github.com/stackblitz/alien-signals/releases/tag/v1.0.0), [v2.0.0](https://github.com/stackblitz/alien-signals/releases/tag/v2.0.0), [v3.2.0](https://github.com/stackblitz/alien-signals/releases/tag/v3.2.0).

- **v0.x (2024):** class-based API (`.get()`/`.set()`), separate Dependency/Subscriber. Early perf PRs: [#10 "Clean up overdesign"](https://github.com/stackblitz/alien-signals/pull/10), [#14 store *value* instead of version number in Link for dirty check](https://github.com/stackblitz/alien-signals/pull/14) (+2–4%), [#16 complex-object bench](https://github.com/stackblitz/alien-signals/pull/16), [#29 flatten IF branches in propagate](https://github.com/stackblitz/alien-signals/pull/29) (~0.5% by removing one bit-op), [#30 simplify checkDirty fallback](https://github.com/stackblitz/alien-signals/pull/30).
- **v1.0 (Jan 2025):** `createReactiveSystem()` decouples the core from the surface API; switched **class-based → function-call API** (`count()` / `count(2)`) — release notes say class API minimized memory but had worse DX; claimed **+12% vs v0.6** on internal bench. [Issue #38](https://github.com/stackblitz/alien-signals/issues/38) reported a ~1.8% regression vs v0.6 in js-reactivity-benchmark; Johnson's debugging method (from that thread): *"modify js-reactivity-benchmark to run only a single test and enable Chrome performance panel line-level timing"*; fixed by v1.0.7 (notably [#50 linked-list stack instead of prevSub reuse in propagate](https://github.com/stackblitz/alien-signals/pull/50), [#51 exit early when a key node fails dirty-check](https://github.com/stackblitz/alien-signals/pull/51)). v1.0.5–1.0.6 flip-flopped the effect queue between array and linked list.
- **v2.0 (Apr 2025, single mega-PR [#61](https://github.com/stackblitz/alien-signals/pull/61)):** merged Subscriber+Dependency into `ReactiveNode`; `propagate` now only marks `Pending` (Dirty applied via `shallowPropagate` after confirmed change) — **deferred signal evaluation** so `src(999); src(10)` never recomputes; `unwatched` callback fires when a node loses all subscribers (no recursive teardown); removed `processEffectNotifications`/`updateDirtyFlag` etc., added `unlink`/`checkDirty`.
- **v2.0.6:** per-cycle version dedup (54fe1b3, above). **v3.0 (breaking, teased in [issue #70](https://github.com/stackblitz/alien-signals/issues/70)):** `createReactiveSystem` moved to `alien-signals/system` entrypoint (treeshaking, [issue #68](https://github.com/stackblitz/alien-signals/issues/68)); removed `pauseTracking`/`resumeTracking` in favor of `setCurrentSub(undefined)` ("for better performance"); removed `startTracking`/`endTracking`; renamed to `getActiveSub`/`setActiveSub`; const-enum inlining fixes for emitted JS ([#88](https://github.com/stackblitz/alien-signals/pull/88), [#89](https://github.com/stackblitz/alien-signals/pull/89)); fast path for first computed read.
- **v3.1:** `trigger()` API for manual dirtying of mutated objects ([#91](https://github.com/stackblitz/alien-signals/pull/91); related: [issue #83 "eager option"](https://github.com/stackblitz/alien-signals/issues/83) was solved in userland instead). **v3.2 (May 2026):** effect cleanup functions (#113); nested-effect LIFO disposal (#116); `innerWrite` parameter to `propagate` + `runDepth` counter (fixes #112 — inner writes blocking future propagation through computed chains); `checkDirty` made resilient to graph mutation during update (#109/#110 — the `return dirty && !!sub.flags` guard); integrated [reactive-framework-test-suite](https://github.com/johnsoncodehk/reactive-framework-test-suite) — **alien-signals is the only library passing all 180 conformance tests** (preact 178, @vue/reactivity 172, anod 166, solid-js 154, mobx 148, TC39 polyfill 145, svelte 131, reactively 114).
- Known open edge case: [issue #123](https://github.com/stackblitz/alien-signals/issues/123) — cyclic computed graphs infinite-loop/OOM; no cycle detection in the hot path by design (DAG assumed).

## 4. Vue 3.6 adoption and what Vue changed

## 5. Performance discussions most relevant to your optimization work

**[Issue #108 — "Linked list vs array-based notify() performance"](https://github.com/stackblitz/alien-signals/issues/108)** is the single most relevant thread. Vilhelm (author of array-based [anod](https://github.com/visj/anod)) benchmarked alien vs anod with mitata including heap: anod wins **Update: wide dense −37% time / −89% heap**, **Update: large web app −24% / −85%**, **Unstable −50%**, **1k signal creation −21% / −75% heap**; alien wins **Diamond +9%**, **CellX 10-layers +12%**, **Update: deep +17%**, **1k computations +23%**. Johnson's reply concedes the structural trade-off explicitly: heap churn in alien's update benchmarks comes from **`{value, prev}` stack frames allocated at each branching point in propagate/checkDirty** — *"This is inherent to linked list traversal — arrays can iterate subscribers with a simple loop. That's the fundamental trade-off: linked lists favor deep propagation and O(1) link/unlink; arrays favor wide fan-out and lower allocation pressure."*

**[Issue #117 — "DCE-safe harness + zero-GC architecture"](https://github.com/stackblitz/alien-signals/issues/117)** — direct evidence about typed-array/data-oriented designs: the [lite-signal](https://github.com/PeshoVurtoleta/lite-signal) author built a pre-allocated, flat-memory-arena, stack-free engine and reports that **an early attempt to build the whole dependency graph on Typed Arrays "turned out slower than expected"**. Cross-benchmarks ([volynetstyle's matrix](https://volynetstyle.github.io/js-reactivity-benchmark/)) show alien-signals beating the zero-GC engine 3–8x on dynamic/topology-heavy workloads (e.g. large-web-app 368ms vs 2924ms), with the conclusion that *"zero-GC and topology scalability are orthogonal dimensions"* — dynamic dep-retracking cost (cursor miss → linear scan) dominates once graphs churn, not allocation.

**Effect-queue representation flip-flops:** [PR #78 "queue effects in linked lists"](https://github.com/stackblitz/alien-signals/pull/78) (motivated by JS-specific array tricks regressing in the Dart port, [medz/alien-signals-dart#20](https://github.com/medz/alien-signals-dart/pull/20)) and [PR #76 eliminate notify recursion](https://github.com/stackblitz/alien-signals/pull/76); [issue #93](https://github.com/stackblitz/alien-signals/issues/93) documents that intrusive linked-list queues (`nextEffect` field) are **~20% faster in Dart AOT but neutral-to-negative in JS**; current JS impl uses the flat array again.

**Rejected simplifications:** [PR #95](https://github.com/stackblitz/alien-signals/pull/95) removed `Link.version`, the `Recursed` flag and `isValidLink` — all tests passed and propagate microbench improved ~5% — but Johnson rejected it: the checks are semantically required (Vue's test suite catches it; alien's own suite didn't at the time). Useful caution: alien's correctness envelope is bigger than its local test suite.

**Internal benchmark:** `benchs/propagate.mjs` (mitata, run with `node --jitless --expose-gc`) builds a w×h grid — w parallel chains of h stacked computeds each ending in an effect — and measures `src(src()+1)`; w,h ∈ {1,10,100}. `--jitless` is used deliberately to measure interpreter-level op counts. `benchs/memoryUsage.mjs` measures per-node memory (from #76 thread: signal 1.17KB→, computed 2.6KB, effect 3.1KB per... reported as totals for their creation loops).

## 6. Why alien-signals is fast vs Reactively / Preact / Vue (mechanism comparison)

- **vs Preact** ([Signal Boosting](https://preactjs.com/blog/signal-boosting/)): Preact pioneered the doubly-linked Node reuse (mark-unused/reuse/purge + list reversal, per-signal + global version numbers, 4-step staleness checks). alien-signals keeps the linked-list layout but replaces version-number comparison with **flag-state propagation (Pending/Dirty) pushed at write time**, so reads don't re-walk dep lists comparing versions; its dep reuse is positional (tail-cursor) rather than `source.node` pointer-stashing, avoiding the post-run list reversal.
- **vs Reactively** ([graph coloring](https://milomg.dev/2022-12-01/reactivity)): same lazy push-pull idea (red=dirty / green=check ≈ Dirty/Pending; `updateIfNecessary` ≈ `checkDirty`; equality-cut via version ≈ `shallowPropagate` upgrade). Differences: Reactively stores deps in **arrays** (`this.sources`) with index-based reconciliation and uses recursion; alien-signals' linked lists give O(1) unlink for dynamic dep sets and its iterative traversal avoids call-stack overhead on deep chains; flag dispatch is branch-flattened bitwise ops.
- **vs Vue 3.4/3.5:** Vue 3.4 (Johnson's #5912) had dirty levels but eager `Dirty` propagation and Dep map objects; Vue 3.5 was Preact-style pull with globalVersion; 3.6 is the alien-signals port. alien's remaining edge vs @vue/reactivity in benchmarks is mostly Vue's ref/Proxy wrapper overhead and scheduler integration, not the core (per Johnson in #44).
- **Benchmarks:** the canonical suite is js-reactivity-benchmark — [transitive-bullshit fork](https://github.com/transitive-bullshit/js-reactivity-benchmark) (referenced by README; alien-signals added Nov 2024 and topping the average-time chart) and the maintained [milomg fork](https://github.com/milomg/js-reactivity-benchmark) (per-test bar charts; alien-signals fastest or near-fastest on avoidablePropagation, deepPropagation, diamond, repeatedObservers, unstable, cellx1000, createComputations; Reactively slightly ahead on some dynamic-graph tests like `4-1000x12 dyn5%` and `25-1000x5`). Both are checked out locally at `/Users/jitl/src/alien-signals-opt/{tb,milomg}-reactivity-benchmark`.

## 8. Implications for our optimization project (synthesis of the community's findings)

1. **The known, admitted bottleneck is wide fan-out + allocation of DFS stack frames** in `propagate`/`checkDirty` (issue #108). An array (or reusable scratch buffer) for subscriber iteration and/or a pooled/preallocated stack would attack exactly the −37%/−89% heap gaps anod demonstrated, at the risk of losing O(1) unlink.
2. **Naïve typed-array/SoA graph storage has already failed once** (issue #117, lite-signal author) — the loss appears to come from dynamic dep-retracking (cursor-miss → linear scan) and JS boxing at the boundaries, not from the layout itself; any SoA design must preserve the tail-cursor O(1) re-track fast path and cheap mid-list unlink (e.g. index-based free-lists).
3. **Dedup and the version/Recursed/isValidLink machinery are load-bearing** (PR #95 rejected); test against Vue core's suite and reactive-framework-test-suite (180 cases), not just alien's own tests, before claiming a faster variant is correct.
4. **Queue representation is language-dependent** (array wins in JS, intrusive linked list wins in Dart AOT — #93/#78); micro-decisions here were empirically flip-flopped multiple times.
5. Johnson's profiling workflow: single-test js-reactivity-benchmark runs + Chrome performance panel line-level timing; internal microbench uses `--jitless` mitata grid benches.
