# Skip / skiplang / SKDB / Noria — Runtime Mechanisms and Lessons for a Signals Library

Research domain: Meta/SkipLabs' Skip ecosystem (Skip framework, skiplang language/runtime, SKStore incremental engine, SKDB reactive SQL DB) plus Noria's partially-stateful dataflow (OSDI 2018). Sources include fetched blog posts, the podcast transcript, the OSDI paper PDF, and **direct source inspection of a fresh clone of github.com/SkipLabs/skip at `/tmp/skiplabs-skip`** (key files cited with absolute paths below).

---

## 1. The Skip ecosystem at a glance

- **Skip framework** ([github.com/SkipLabs/skip](https://github.com/SkipLabs/skip), [skiplabs.io/blog/why-skip](https://skiplabs.io/blog/why-skip)): TypeScript-facing framework for reactive backend services. You "write your program as if time is frozen"; the runtime turns it into a live streaming system. Two collection types: **eager** (always up-to-date, recomputed on source update) and **lazy** ("compute and cache results only when needed … automatically evict data from the cache to save memory"). The reactive graph lives in a **wasm (32-bit) or native skiplang runtime** (`@skipruntime/wasm` / `@skipruntime/native`); TS mappers are deterministic, effect-free functions called across the boundary; collections hold JSON keys/values ([docs introduction](https://skiplabs.io/docs), local copy `/tmp/skiplabs-skip/www/docs/introduction.md`). Per-request **resource instances** extend the shared graph dynamically and are "garbage collected by the Skip framework after a period of inactivity" (`/tmp/skiplabs-skip/www/docs/resources.md`).
- **skiplang** ([skiplang.com](https://skiplang.com/), original repo [github.com/skiplang/skip](https://github.com/skiplang/skip)): AOT-compiled (LLVM) language whose type system **statically tracks side effects and immutability** ("frozen" types, `~>` frozen lambdas), which is what makes safe memoization, safe parallelism (`tabulate` over a folly CPUThreadPoolExecutor with an atomic work-stealing counter; "race conditions impossible" because lambdas may only capture frozen objects — [parallelism post](https://skiplang.com/blog/2017/11/30/parallelism.html)), and cheap GC possible.
- **SKStore** (a.k.a. SKFS): the incremental-computation engine written in skiplang, in `/tmp/skiplabs-skip/skiplang/prelude/src/skstore/` with a design doc at `/tmp/skiplabs-skip/skiplang/docs/SKStore.md`. This is the concrete machine underneath both the Skip framework and SKDB.
- **SKDB** ([dbdb.io/db/skdb](https://dbdb.io/db/skdb), sources under `/tmp/skiplabs-skip/sql/`): reactive SQL database; "your queries are in fact a graph of computation in this reactive file system" — SQL views compiled to SKStore directories. MVCC, SQLite-inspired, row-oriented custom storage.

## 2. skiplang runtime: heap layout and reclamation

This is the most transferable part for data-layout thinking. Sources: [Julien Verlaguet's "Memory Model For RP"](https://jverlaguet.com/reactive%20programming/2024-06-23-memory_model.html), [podcast transcript](https://podscripts.co/podcasts/materialized-view-podcast/skdb-and-reactive-databases-with-julien-verlaguet), and runtime C sources in `/tmp/skiplabs-skip/skiplang/prelude/runtime/`.

**Two-heap scheme (obstack + shared interned heap):**
- Each recomputation allocates into a thread-local **obstack**: a bump allocator over a linked list of pages (`obstack.c`; page header = prev-page pointer + size, oversized allocations get their own exactly-sized page).
- Results that survive a computation are **interned** (deep-copied, hash-consed) into a **Shared Heap** with per-object **reference counts** (`intern.c`: `shallow_intern`, `sk_incr_ref_count`, `SKIP_intern`). "Long-lived objects are stored in the so-called Shared Heap, using a reference counting scheme"; short-lived garbage is handled by "a tracing collector for the objects that were allocated during the update" that "stops whenever it hits an object that lives in the Shared Heap" — so **"the GC will never visit more objects than what was allocated during an update"** and "typically only visits objects that live in a fast cache (L1/L2)". SKStore.md states the invariant: **"garbage collection time is a function of the size of the update, not of the size of the heap."**
- The type system "guarantees statically that those objects are immutable and do not have cycles" — refcounting suffices, and deep `native_eq` equality is safe (`native_eq.c`).
- **The persistent heap IS the database**: `palloc.c` mmaps a fixed-address region (`BOTTOM_ADDR 0x0000001000000000`, default capacity 16 GB) so interned pointers stay valid across processes; in wasm32 the runtime tracks **dirty pages** (`sk_persistent_write`, `sk_dirty_pages_stack` in `intern.c`) to flush incrementally.
- Why not Rust: ownership fights shared memo caches — "the very nature of Rust's ownership model is such that shared state is avoided, making traditional caching difficult"; you end up "working in a manner resembling a purely functional language, but with the overhead of atomic reference counting" ([Rust for RP](https://jverlaguet.com/reactive%20programming/2024-06-24-rust_for_rp.html)).

**Old skiplang (2018) memoization runtime** ([Reactivity Overview](https://skiplang.com/blog/2017/01/04/how-memoization-works.html)) — the pre-SKStore design, closest to a signals runtime:
- Program split into a memoizing **Core** (parallel "reader tasks") and a single non-memoizing **Mutator** that changes **Cells**. "When a Core function reads a Cell … the runtime automatically creates an edge in the dependency graph between that Cell and the function's memoized result value"; likewise for memoized-calls-memoized (even through non-memoized frames).
- Graph is "related to ADAPTON's Demanded Computation Graph (DCG), but extended with **multiversion concurrency control (MVCC)** and **incremental coarsening of dependency information to reclaim memory**" (coarsening ≈ LRU-driven merging of fine-grained dep records; trading recompute granularity for space).
- **Revision lists**: "Each Cell and each memoization cache entry records not just a single value, but a linked list of 'revisions', each … a value annotated with a non-overlapping [begin, end) transaction ID 'lifespan'." Commits propagate invalidations, then bump a global transaction ID; readers latch the ID at start and only see values "as of" it; "as old tasks exit, revisions corresponding to transaction IDs that no tasks care about any more get garbage collected, keeping the lists very short in practice." No global locks.
- Async memoization ([Async/Await post](https://skiplang.com/blog/2018/07/18/async-await.html)): each memoized invocation gets its own **Process** ("a private GC heap, an event queue and some reactive graph construction state"); a second caller of an in-flight memoized computation "will immediately get back an Awaitable" instead of blocking. Awaitable size = max live-variable set at any single `await`, with graph-coloring register allocation for suspended locals.

## 3. SKStore engine internals (from source)

The current engine models everything as **directories** (collections) of key → `Array<File>` values. All file paths below are in the clone.

**Dependency representation** (`Context.sk`):
- An **`Arrow(parentName, childName)`** is a map edge between directories. An **`ArrowKey{parentName, childName, key}`** identifies one keyed computation instance ("the map from A to B on key K"; for lazy dirs parent == child).
- Reads are tracked in a global two-way index, `value class Deps`:
  - forward: `SortedMap<Path, SortedSet<ArrowKey>>` — for a read `Path(dirName, key)`, the set of computations that read it;
  - inverse: `HashMap<DirName, SortedSet<Path>>` — per child directory, every path it reads (used to drop all edges when a directory dies, `removeDir`).
- During a mapper run the context swaps in an empty `reads: SortedSet<Path>`; every `getArray` calls `context.addRead(path)`. Afterward (`EagerDir.update`, ~line 1560 of `EagerDir.sk`): `updateNewDeps(arrow, reads)` adds new edges and `removeDeps(arrow, oldReads.filter(r -> !reads.contains(r)))` removes stale ones — i.e. **set-diff of read-sets per re-run**, the same job alien-signals does by relinking its doubly-linked Link lists in place.
- Per source key, the child stores `old: SortedMap<Path, MInfo>` — what that computation produced last time: `MInfoEmpty | MInfoSingle(Key) | MInfoFull(keys, newDirs, reads)` (note the compact one-key special case as a memory optimization). Used to delete keys no longer produced, tear down sub-directories created last run, and diff reads.
- **Aggregate dependency keys**: reading a directory's size, emptiness, or key-set records a dependency on synthetic paths `Path::sizeTag / isEmptyTag / filesTag / dirTag` (`Path.sk`), and `writeEntry` only invalidates `sizeTag` when `totalSize` actually changed and `isEmptyTag` only when emptiness flipped — fine-grained derived-property dependencies instead of whole-collection edges.

**Dirty propagation and scheduling** (`Context.sk` `updateWithStatus`, `EagerDir.writeEntry`):
- A write does **early cutoff first**: `if (native_eq(rvalues, oldValues) == 0) return` — value-level equality on immutable interned data at every node boundary.
- It then records the delta at the current **Tick**, marks the key dirty (`context.addDirty`), notifies keyed readers via `updateDirtyReaders(path)` (lookup in `Deps`), and schedules `Arrow(this, child)` for every child directory via `addToUpdate(childTimeStack, parentTime, arrow)`.
- `toUpdate: SortedMap<(TimeStack, Time), Arrow>` is used **as a priority queue**; the update loop repeatedly `removeMin()`s and re-runs only dirty keys of that edge. A **`TimeStack`** is the stack of creation `Time`s of a directory (nested creation contexts extend the parent's stack). Since a derived directory is always created after its sources, ordering by TimeStack processes ancestors before descendants — **topological order derived from creation time, no heights, no re-entrancy**. Lazy self-arrows are enqueued with negated TimeStacks so invalidation of lazy caches sorts before eager recomputation. One pass over the queue = one transaction; at the end `tick.next()`.
- Two distinct clocks: `Time` (directory creation order; drives scheduling) and `Tick` (global commit counter; drives versioning/diffs).

**Versioned storage layout — flat sorted arrays + tick-annotated AVL overlay** (`FixedDir.sk`, `DMap.sk`, `EagerDir.sk`):
- Every eager directory = **`fixedData`**: an immutable **flat sorted array** of `FixedRow(key, value, source: Path, tag: TickRange)` searched by binary search (`findFirstBy`/`findAllBy`), plus **`data`**: a `DMap` overlay of changes since the array was built. A value's identity is (key, source) where `source` is provenance — which parent computation wrote it — so multiple mappers can write the same key and be updated independently.
- **`DMap`** is an AVL tree where **every node carries `TickRange{max, current}`**: `current` = tick of that entry's last change, `max` = max tick in the subtree. "We can walk the tree and prune all the parts that are 'too old'" — `getChangesAfter(tick)` visits only subtrees with `max > tick`, so **diff-since-tick costs O(log n + |changes|)**. This powers cheap re-subscription ("quickly retrieve a diff of any eager directory between the present and any point in the past", SKStore.md) and crash/disconnect recovery.
- Deletion = writing an empty array (a **tomb**); tombs are kept so diffs can report removals, and are "collected automatically."
- **Compaction ("purge")**: when the delta overlay grows as large as the base array (`data.nbrEntries >= fixedData.size()`), the dir is queued for purge; `purge(limit: Tick)` flattens fixed+delta into a fresh sorted array and drops tombs older than `limit` (`flattenData`, `purgeOld`). Default history horizon: `purgeFrom = tick - 1000` (Context.sk:421); slice-index churn is compacted after `PURGE_LIMIT = 30` mutations. If a subscriber asks for changes older than `tombLimit`, `getChangesAfter` returns `(true, allKeys)` — i.e. "history gone, do a full rebuild" (`!rebuild` in the replication protocol).
- **Incremental aggregates**: an optional `Reducer` per directory keeps per-key state updated by `reducer.update(state, oldValues, newValues)`; if the reducer can't apply the delta (returns `None`, e.g. non-invertible aggregates like min after removing the min) it falls back to `reducer.init(fullScan)` (`EagerDir.writeEntry`).

**Lazy directories and eviction** (`LazyDir.sk`, `Handle.sk`, `Context.updateLazyGets`):
- Lazy values are `LazyResult = LDefined(values, reads) | LRemoved | LCycle | LAbsent` — `LCycle` is an in-progress marker giving cycle detection (throws `CyclicExn`), like a signals library's "computing" flag.
- Eviction is a **sliding window over update rounds**: each round's read key-set (`lazyGets`) is pushed onto a queue of length `lazyCapacity` (default 10); when a round falls off, per-key refcounts (`lazyGetsRefCount`) decrement and zero-ref keys are dropped from collecting lazy dirs. So: "values are automatically evicted if they are not used often enough. The cache eviction policy is programmable" (SKStore.md). A cheap, allocation-light generational approximation of LRU keyed on propagation rounds rather than wall time.

**Transactions/concurrency** ([Concurrency in SKDB](https://jverlaguet.com/reactive%20programming/2024-08-05-concurrency_skdb.html), podcast): begin = "a pointer to your version of the data" (snapshot, no copy); the transaction builds its result off-thread; at commit "because the whole system is reactive … only update what has changed between the moment you started and the moment where you're trying to commit," under a global lock held "for a very, very brief amount of time." Guarantees (SKStore.md): transactional all-or-nothing updates; concurrent readers/writers block only at commit.

**Performance claims** (podcast): write propagation is "log n on the size of the table, not O(n). You don't want to have to scan anything when you do a write." Versus dedicated streaming engines (differential dataflow etc.): "we probably would be 2x slower, maybe 5x slower" on their home turf, but memoization wins when there are **many overlapping cached queries** (each new query reuses the shared graph) — see [Streaming vs Memoization](https://jverlaguet.com/reactive%20programming/2024-08-10-streaming_vs_memoization.html): eager dirs "update quickly but take up more memory," lazy dirs "update slower … and use less memory because they can 'forget' values"; build indexes eagerly, transitive closures lazily.

## 4. Noria: partially-stateful dataflow (OSDI 2018)

Paper: [pdos.csail.mit.edu/papers/noria:osdi18.pdf](https://pdos.csail.mit.edu/papers/noria:osdi18.pdf) (read directly, §3–§8); good secondaries: [the morning paper](https://blog.acolyer.org/2018/10/29/noria-dynamic-partially-stateful-data-flow-for-high-performance-web-applications/), [micahlerner.com](https://www.micahlerner.com/2021/03/28/noria-dynamic.html). 45k lines of Rust; base tables in RocksDB (one column family per table; persistence costs ~5% write throughput).

**Model**: DAG of relational operators; base tables at roots, external views at leaves. Writes become **signed deltas** (positive/negative records; "negative updates hold exactly the same values as the positives they revoke and thus follow the same data-flow paths"). External views are **double-buffered hash tables** — dataflow worker writes one map, readers read the other, atomic pointer swap publishes; "it can improve read throughput by 10× over a single-buffered hash table with bucket-level locks" under skew. At most one worker executes updates for an operator at a time; operators in a chain are scheduled as a unit to cut queueing and cross-core movement.

**Partial state & upqueries — demand-driven pull inside a push system** (§4):
- Operators keep only a subset of their state; missing entries are `⊥`. On a read miss (or a join needing the other side), the operator issues an **upquery**: "requests for matching records from stateful ancestors"; misses recurse "all the way to the base tables if necessary." Responses are **snapshots of operator state**, flow down the single path back to the querier without forking, and "commute neither with each other nor with previous updates" — hence strict ordering rules.
- Correctness invariants over each state entry e (with T_e = inputs so far, S_e = inputs actually reflected, D_e = key-descendant entries): **update completeness** (if S_e ≠ ⊥, missed updates are in flight or an eviction notice is); **no spurious/duplicate updates** (S_e ⊆ T_e); **descendant eviction** (if S_e = ⊥ then every key-descendant is evicted or being evicted); **eventual consistency**.
- **Join upqueries**: scoped so "no updates are in flight between the upstream stateful operator and the join when a join upquery occurs: each join upquery is scoped to an operator chain processed by a single thread." Trade-off: joins store no duplicate windowed state (unlike stream processors' windowed caches), but chains containing joins can't be split across threads without duplicating state. Branching subgraphs (unions): recursive upqueries execute independently, then the final result is computed with join upqueries "once the recursive upqueries complete."
- **Eviction**: under memory pressure, **eviction notices "flow forward along the update data-flow path"**; downstream operators drop updates targeting evicted entries "without further processing or forwarding" (that's where the memory/CPU saving comes from — evicted keys stop costing write amplification). If a join upquery hits evicted state during update processing, Noria "chooses to drop the update and evict dependent entries downstream" (statically analyzing the graph to compute the needed eviction notices) "to avoid blocking the write path while filling in the missing state."
- **Materialize vs recompute policy** (§4.5): "Noria makes state partial whenever it can service upqueries using efficient index lookups. If Noria would have to scan the full state of an upstream operator … Noria disables partial state for that operator" (e.g., unparameterized views like "top 20 stories by vote count"). Also, "partial-state operators cannot have full-state descendants" (descendant-eviction invariant). Indexes come from **index obligations** implied by operator semantics (e.g., vote-count aggregation needs a story_id index). Partial operators "start out fully evicted and are lazily populated by upqueries."
- **Live transitions** (§5): new queries are merged into one joint dataflow (query-graph matching → fine-grained IR → splice at deepest common subexpression). New partial operators start empty and warm via read-driven upqueries ("amortizing the bootstrapping work over time"); new full-state operators are "inactive" until one special big upquery for all keys backfills them. Reuse + partial state make transitions near-instant (Fig 10: without them, full copies stall writes for ~25s).

**Numbers**: Lobsters app: 5× MariaDB throughput (5,000 vs 1,000 pages/s) with "natural" queries; subset benchmark 14M req/s (95/5 read/write, Zipf, 4 shards, one 16-vCPU server), beating even memcached-only; 2M req/s at 50/50. vs **DBToaster**: 240k single-record writes/s fully-populated and **1M writes/s fully-evicted** (evicted state = dropped work) vs DBToaster's 520k; memory 6.2 GB vs 17 GB. vs **differential dataflow**: comparable at 1–2 machines, Noria scales linearly to 30M req/s at 10 machines while DD tails off ~20M due to progress-tracking coordination. **State size**: Lobsters dataflow = 235 operators, 60 stateful; full state 789 MB (8× the 137 MB base tables); with partial state only 73 MB is essential (9%), the rest evictable; working set 525 MB at production load; at 10× scale, sub-100ms latency needs a 2.6 GB working set (38% of 7 GB total; ~3× base tables). Consistency: eventual; requires deterministic operators and **commutative multi-ancestor operators**, no update/upquery races, no same-path reordering.

## 5. Lessons for alien-signals

Dependency representation:
1. **Read-set diffing is the universal pattern** — Skip re-runs a keyed computation, collects `reads`, and add/removes edges by set difference; alien-signals' in-place relink of its Link list is a faster specialization of the same thing. Skip's `Deps` two-way index (forward: source→readers; inverse: reader→sources) mirrors alien-signals' two doubly-linked lists per Link; the inverse index existing *only* to support bulk teardown (`removeDir`) suggests: if you restructure Links, keep whichever direction supports (a) invalidation fan-out and (b) O(outgoing) disposal, and consider a compact array-of-source-ids per subscriber (rebuilt by diffing) instead of linked list surgery — Skip demonstrates correctness of that model at scale.
2. **Flat sorted array + small delta overlay + periodic flatten** (FixedDir/DMap + `purge` when delta ≥ base, PURGE_LIMIT=30) is Skip's core data-layout trick: immutable cache-friendly arrays with binary search for the cold majority, a tree only for recent changes, amortized compaction. For a struct-of-arrays alien-signals port: store stable dependency edges/node metadata in flat typed arrays rebuilt lazily, keep churn in a small overlay, compact when overlay ≈ base.
3. **Provenance-tagged values** (`FixedRow.source`) let multiple producers write one key and be invalidated independently — relevant to any keyed/collection-signal extension.
4. **Derived-property dependency keys** (`sizeTag`/`isEmptyTag`/`filesTag`, invalidated only when size/emptiness actually changes) — cheap built-in "selector" nodes that cut invalidation fan-out without allocating real computed nodes.

Versioning & dirtiness:
5. **Two clocks**: creation-order `Time` for scheduling and commit `Tick` for versioning. `toUpdate` keyed by TimeStack is a **priority queue in creation order = topological order for free** (derived nodes are created after their sources; nested creation extends the stack; lazy invalidations sort negative/first). For alien-signals this suggests an iterative, queue-driven propagation ordered by a per-node creation index instead of recursive flag walks — with the caveat that in a general signals API a computed *can* read a node created later, so a creation index is only a valid priority when the API constrains construction order (Skip enforces it structurally) or when paired with on-demand reordering.
6. **Hierarchical version stamps**: DMap nodes carry `TickRange{current, max-of-subtree}` so "diff since tick T" prunes clean subtrees — the tree-shaped generalization of alien-signals' per-link version counters, and the mechanism behind O(log n + changes) resubscription/recovery. A signals library could maintain max-child-version summaries at fan-in nodes to skip whole clean regions during pull-validation.
7. **Early cutoff by deep value equality on immutable data at every node boundary** (`native_eq` before any dirty marking) — more aggressive than alien-signals' reference/version check; viable because values are frozen and interned (hash-consing makes equal things pointer-equal or at least cheap to compare).

Memory & reclamation:
8. **Arena-per-update + refcounted shared heap**: allocate recompute scratch in a bump region discarded wholesale; intern only survivors; "GC time is a function of the size of the update, not of the size of the heap." The JS analog is zero-allocation recompute paths and pooled Link records (which alien-signals already pursues); for a wasm/typed-array design, this is a blueprint: per-propagation arena, survivors copied into a persistent SoA store.
9. **Eviction as a first-class, programmable policy**: Skip's lazy dirs use a sliding window of N update rounds with per-key refcounts (default 10) — an O(1)-per-touch LRU approximation keyed on propagation rounds; Skip also reclaims dependency metadata itself ("incremental coarsening of dependency information") and resource instances by inactivity timeout. alien-signals' unwatched-computed handling could generalize to keyed memo families with round-based eviction.
10. **History with a horizon**: tombstones + `tombLimit` (default: 1000 ticks) with a "full rebuild" fallback when a consumer is too far behind — the right shape for any replay/subscription API.

Combining eager push with lazy pull (Noria ↔ push-pull signals):
11. Noria is exactly "push-based flags propagation + pull-based recompute" at database scale: deltas push down; **upqueries pull up on demand**, and the hard part is ordering — upquery responses are snapshots that don't commute with in-flight updates, solved by scoping pulls to single-threaded chains and dropping updates to missing state. The signals moral: when a pull (validation of a stale computed) races a push, you need a discipline equivalent to alien-signals' flags; Noria's is "drop pushes into evicted state; never block the write path on a pull."
12. **Evicted = unsubscribed, formally**: Noria's *descendant eviction* invariant (a cold entry's key-descendants must also be cold) is the correctness rule behind alien-signals' unwatched-computed trees, generalized to keyed state; its eviction notices flowing forward are push-domain "this subtree went cold" messages that let intermediate nodes skip work entirely.
13. **Materialize vs recompute heuristic**: cache (materialize) only what can be refilled by an *index lookup* (O(1)/O(log n) pull); never partial-cache anything whose refill requires a full upstream scan — make it either always-eager or always-recompute. Applied to signals: memoize keyed computeds only when re-derivation on miss is cheap and keyed; keep unkeyed aggregates eager.
14. **Warm-up by demand**: new partial operators start empty and fill via read-driven pulls, amortizing graph transitions — the same trick as lazily initializing computed values on first read, extended to whole subgraphs.
15. **Benchmark framing**: eviction can *increase* write throughput (Noria: 1M writes/s fully-evicted vs 240k fully-populated) because dirty pushes to cold state are dropped — a measurable argument for aggressive cold-node pruning in signals propagation.
