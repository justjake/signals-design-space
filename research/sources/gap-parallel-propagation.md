# Parallel & Concurrent Change Propagation in Reactive/Signal Graphs — and What alien-signals Could Exploit with Typed-Array (SAB) Storage

## 0. Executive summary

- **FrameSweep/MV-RP (REScala, OOPSLA 2018)** is the definitive design for *glitch-free concurrent propagation*: per-node **version histories** + conservative 2PL "framing" traversal that plants **placeholders**, MVCC for reads, and **retrofitting** to repair histories when dynamic edges appear "in the past". It gets pipeline parallelism between overlapping transactions, but costs **25% single-thread overhead** and needs **~160 µs of user compute per update** to scale to 16 threads. That is the realism anchor: alien-signals updates are typically ≪10 µs, so parallelizing *one* propagation wave across JS workers will not pay; the exploitable ideas are pipelining *independent* transactions, partition ownership, and version-counter (seqlock/MVCC-lite) reads.
- **Salsa** shows the practical minimum machinery for a *multi-reader* signal graph: a global **revision counter array indexed by durability**, per-memo `verified_at`/`changed_at` stamps, a **SyncTable** (`FxHashMap<Id, SyncState>`) so two threads never compute the same query twice (block, don't duplicate), a mutex-protected **thread dependency DAG** for cross-thread deadlock detection, and cancellation via an `AtomicBool` checked cooperatively and thrown as `Cancelled::{Local, PendingWrite, PropagatedPanic}` via `panic::resume_unwind`. All of this maps cleanly onto typed-array storage: revision = one atomic `u32`, claim = one CAS per node.
- **Leptos `reactive_graph`** is the only production multithreaded *signal* graph surveyed: per-node `Arc<RwLock<T>>` value + `Arc<RwLock<SubscriberSet>>`, `Weak` subscriber refs, thread-local observer stack, tri-state `Clean/Check/Dirty` push-pull. Its open PR #3681 documents the exact pitfall: asymmetric lock behavior (reads panic under contention, writes block) → panic-vs-deadlock tradeoffs. Locking is *per node, two locks per node* — no global lock, no glitch-freedom guarantee across threads.
- **PSAC (SPAA 2021)** proves parallel change propagation *can* win big — but only for **batch** updates: 22–39× propagation speedups on 64 hyperthreads for large batches, work savings up to **819,600×** for single-element updates, yet essentially **zero parallel benefit at batch size k=1**, and 1.7–7.3× tracking overhead + 1–7× memory overhead.
- **JS substrate measurements (mine, Node v24.16.0, Apple Silicon)**: uncontended `Atomics.load/store/add/or` cost **4.6–5.0 ns vs 0.33–0.53 ns** plain typed-array ops (~10×); `Atomics.compareExchange` 6.3 ns; `Atomics.wait/notify` cross-thread round trip **5.5 µs**; `postMessage` round trip **7.9 µs** (Node; browsers are slower and structured-clone cost scales with payload — Surma's data: ~100 KiB fits a 100 ms budget on the slowest phones, only ~10 KiB fits a 16 ms frame). Partytown reports its SAB+Atomics mode is **10× faster** than its service-worker fallback.
- **Nobody has shipped a JS signal graph over SharedArrayBuffer** — searches surface task queues, ring buffers, cross-tab sync recipes, and Partytown's API proxying, but no reactive dependency graph in shared memory. The niche is open; the constraints (only numbers/flags/topology/versions can be shared, not JS object values) dictate the shape sketched in §6.

## 1. FrameSweep / MV-RP — "Thread-Safe Reactive Programming" (Drechsler, Mogk, Salvaneschi, Mezini, OOPSLA 2018)

Paper: [PDF](https://programming-group.com/assets/pdf/papers/2018-Thread-safe-reactive-programming.pdf), [ACM DOI 10.1145/3276477](https://dl.acm.org/doi/10.1145/3276477). The OOPSLA paper names the scheduler **MV-RP** (Multi-Version RP); Drechsler's dissertation ([TUprints](https://tuprints.ulb.tu-darmstadt.de/server/api/core/bitstreams/a74d19a0-de64-49dc-a75f-50d35e48a679/content), [d-nb mirror](https://d-nb.info/1223095517/34)) develops it as **FrameSweep**, "the first algorithm to provide synchronous propagation semantics for concurrent change propagation over dynamic distributed dataflow graphs."

**Goal/guarantee:** abort-free **strict serializability** for all user-visible operations (`now`, `update(...)`, `after`, `before`, `depend`), with every update's reevaluations executing in *some* glitch-free order (Theorem 1, formally proven via multiversion serialization graph theory). Abort-freedom is non-negotiable because reactive computations run side effects that cannot be rolled back — this rules out STM-style optimistic execution.

**Three intertwined techniques over one data structure** — the per-node **version history**:
- Every DG node keeps a list of *node versions*, one per transaction that touched it, sorted by a **stored serialization graph** (a DAG over transactions maintained at runtime with an atomic "insert edge between two transaction nodes unless a path already exists" operation).
- **C2PL framing phase:** before executing, a transaction traverses the DG from all changed inputs and plants a **placeholder version** on every reachable node. Placeholders double as (a) planned-reevaluation markers and (b) a FIFO queue for the node's exclusive lock: only the transaction holding the *first* placeholder on a node may reevaluate it.
- **Switch to executing:** allowed once (1) all placeholders are planted and (2) all predecessor transactions in the serialization graph have themselves switched to executing (prevents missing a predecessor's planned write).
- **Pipelined propagation:** a reevaluation runs as soon as the transaction's placeholder is first on that node; on completion the placeholder is *cleared* (replaced by a real value version), which implicitly unlocks the node for the next transaction. This eager per-node unlocking is what gives **pipeline parallelism**: multiple transactions progress simultaneously through overlapping DG regions, like instruction pipelining.
- **MVCC for reads:** `s.now` and glitch-free parameter reads can execute "in the past" by returning an older version from the backlog; reads "in the future" block until C2PL releases that version. Writes can't be done in the past, hence the C2PL/MVCC split.
- **Retrofitting (dynamic edges):** when a reevaluation discovers or drops an edge (`n.discover(d)` / `n.drop(d)`), the write lands mid-history (serialization time ≠ real time). Retrofitting repairs by: (1) for every later-ordered written/placeholder version on the source node, planting placeholders on all nodes reachable from the sink; (2) for every later *written* version, enqueueing the sink for reevaluation in that transaction. Drops reverse both. Proven to preserve glitch freedom because the sink's first placeholder still belongs to the retrofitting transaction, so no later transaction can have observed the stale topology.
- **GC:** a transaction is *complete* only when it and all serialization-graph predecessors are complete; a newer written version makes all older versions in that history deletable (eager traversal on completion, or batched).

**Evaluation (dual Xeon E5-2670, 2×8 cores, JMH):**
- Baselines: **G-Lock** (global lock), **Handcrafted** (per-application manual fine-grained locking), **STM-RP** (ScalaSTM; not abort-free).
- Topology microbenchmarks (fan-in, fan-out, chain, grid; 16 work nodes): MV-RP scales to just under 16 threads *iff* user computation ≈ **160 µs per update** (~10 µs/node). All topologies pipeline down to single-node bottlenecks; STM only scales on one lucky topology; Handcrafted can't pipeline at all.
- Cheap updates (~6.5 µs/update, philosophers, extreme contention): single-threaded overhead **-25% (MV-RP)**, **-55% (STM)** vs G-Lock ≈75 op/ms; MV-RP scales to ~9 threads then decays, ending **5–30% below G-Lock** — the stated lower bound of usability.
- High contention (16 philosophers, no bottleneck chain): MV-RP overhead 20% single-threaded, **beats G-Lock and Handcrafted from 3 threads on** (pipeline parallelism blocks threads only on individual nodes, only as long as necessary; Handcrafted queues up threads even for non-interacting updates).
- Low contention (64 philosophers): MV-RP scales ~2× better than high contention, **outperforms Handcrafted by ~10% at 16 threads** despite starting 20% lower; STM ≈60% of MV-RP.
- Contention cost at 16 threads: low contention −35%/−25%/−15% (Handcrafted/MV-RP/STM); high −75%/−60%/−45%.
- Dynamic edge changes + retrofitting: **negligible** overhead (dominated by the cost of dep collection/compare that dynamic dependencies require anyway).
- "Universe" case study: existing REScala app parallelized with **zero RP-code changes** (just a parallel task list in imperative code), throughput scaled with workers (Fig. 12; second phase parallelizes, first doesn't — Amdahl-shaped curve).
- Scalability depends on only two dimensions: **conflict density** and **user-computation cost**; DG topology and dynamic-edge conflicts don't matter.

**Transferable ideas for alien-signals:** placeholder-as-queued-lock = one CAS-able word per node; framing = your existing `propagate()` push phase, reused as lock acquisition; pipeline parallelism between *transactions*, not within one; version histories are overkill unless you need concurrent writers with strict serializability.

## 2. Salsa's parallel query execution (beyond red-green)

Sources: [book: cycles/plumbing](https://salsa-rs.github.io/salsa/plumbing/cycles.html), [book: fetch](https://salsa-rs.github.io/salsa/plumbing/fetch.html), [src/function/sync.rs](https://github.com/salsa-rs/salsa/blob/master/src/function/sync.rs), [src/runtime.rs](https://github.com/salsa-rs/salsa/blob/master/src/runtime.rs), [src/cancelled.rs](https://github.com/salsa-rs/salsa/blob/master/src/cancelled.rs), [parallel tests](https://github.com/salsa-rs/salsa/blob/3dc4539c7c34cb12b5d4d1bb0706324cfcaaa7ae/tests/parallel/cancellation.rs), [DeepWiki overview](https://deepwiki.com/salsa-rs/salsa/4-advanced-features).

- **Revision counters:** `Runtime` holds `revisions: [Revision; Durability::LEN]` — the current revision plus a "last changed at" revision *per durability tier*. Writes call `report_tracked_write(durability)` bumping all tiers ≤ that durability; `new_revision()` increments the master counter and clears the cancellation flag, only at quiescence. Memo validation is then O(1) in the common case: memo verified in current revision → return; else *shallow verify* — if no input at ≤ the memo's min-durability tier changed since `verified_at`, mark verified; else *deep verify* via `maybe_changed_after` per dependency; **backdating** keeps `changed_at` old when recomputation produces an equal value (their equivalent of alien-signals' `shallowPropagate` early-cut, but pull-side).
- **Same-query dedup (SyncTable):** per query family, a `FxHashMap<Id, SyncState>`; `try_claim` either inserts `SyncState { owner: SyncOwner::Thread(ThreadId), anyone_waiting, is_transfer_target, claimed_twice }` and returns `ClaimResult::Claimed(guard)`, or returns `ClaimResult::Running` → the caller sets `anyone_waiting = true` and blocks via `runtime.block()`, or `ClaimResult::Cycle`. `anyone_waiting` lets the release path skip lock acquisition when nobody queued. Ownership can be **transferred** (`SyncOwner::Transferred`, `ReleaseMode::TransferTo(DatabaseKeyIndex)`) to hand a computed-under lock directly to a blocked dependent.
- **Cross-thread deadlock detection:** a `Mutex<DependencyGraph>` keyed by `ThreadId`; before recording edge T1→T2 (T1 blocks on T2), it checks `depends_on(T2, T1)`; a path back means a cross-thread cycle → invoke the tracked function's cycle strategy instead of deadlocking. While blocked, the thread's query stack is parked in the shared graph and recovered on wake. `WaitResult ∈ {Completed, Panicked, Cancelled}`; a panicking owner runs `release_panicking()` so waiters unwind with `Cancelled::PropagatedPanic`.
- **Cancellation:** `revision_cancelled: AtomicBool` (Release store in `set_cancellation_flag`, Acquire load), set when a writer wants a new revision; long queries cooperatively check and `Cancelled::PendingWrite.throw()` — implemented as `panic::resume_unwind(Box::new(self))` (skips the panic hook/backtrace), caught at the API boundary by `Cancelled::catch` (downcast, else re-unwind). An `AtomicU8` cancellation counter guards overflow.

**Transferable ideas:** durability-tiered revision counters collapse most validation to one atomic load — in an SAB world this is *one shared `u32` per tier*; the claim-word-per-node CAS pattern is exactly how workers should dedup recomputation; cancellation-on-write is how a main thread can invalidate worker pulls mid-flight (flag word + cooperative checks).

## 3. Leptos `reactive_graph` — production multithreaded signal graph

Sources: [graph/node.rs](https://github.com/leptos-rs/leptos/blob/main/reactive_graph/src/graph/node.rs), [signal/arc_rw.rs](https://github.com/leptos-rs/leptos/blob/main/reactive_graph/src/signal/arc_rw.rs), [graph/subscriber.rs](https://github.com/leptos-rs/leptos/blob/main/reactive_graph/src/graph/subscriber.rs), [book appendix: reactive graph](https://book.leptos.dev/appendix_reactive_graph.html), [issue #832 (Send/Sync migration)](https://github.com/leptos-rs/leptos/issues/832), [PR #3681 (parallel access panics)](https://github.com/leptos-rs/leptos/pull/3681), [docs.rs leptos::reactive](https://docs.rs/leptos/latest/leptos/reactive/index.html).

- **Why Send/Sync at all:** not for parallel propagation — for **Tokio multi-threaded SSR** (Axum). In the browser it runs single-threaded; JS types are `!Send` so `Local*` signal variants exist for them.
- **Locking granularity:** two `std::sync::RwLock`s per signal — `ArcRwSignal<T> { value: Arc<RwLock<T>>, inner: Arc<RwLock<SubscriberSet>> }`. No global lock, no transaction layer. Writes take the value write-lock; the `WriteGuard` notifies subscribers on drop.
- **Graph shape:** `AnySubscriber(usize, Weak<dyn Subscriber + Send + Sync>)` — subscriber sets hold **weak** refs (dead nodes skipped on notify); observers are a **thread-local** `RefCell<Option<ObserverState>>` (dependency tracking never crosses threads); tri-state `ReactiveNodeState { Clean, Check, Dirty }` with `mark_dirty` / `mark_check` / `mark_subscribers_check` / `update_if_necessary` — the same MobX/Reactively-family push-pull coloring alien-signals uses (`Dirty`/`Pending` ≈ Dirty/Check).
- **Concurrency semantics = "std::sync::RwLock semantics", nothing more.** PR #3681 documents the sharp edge: reads use a `try` pattern and **panic under contention** ("reactive value already disposed"), writes **block** (can deadlock); options discussed — `try_get()`, always-block (trade panic for deadlock), `.take()` in guards. There is **no cross-thread glitch-freedom**: a thread can read mid-propagation state; leptos accepts this (effects are async-scheduled, UI converges).

**Lesson:** per-node RwLocks + weak subscriber sets is the cheap route to "thread-safe enough" and would map to per-node seqlock words in SAB — but it deliberately gives up the consistency FrameSweep pays 20–25% for.

## 4. PSAC — Efficient Parallel Self-Adjusting Computation (Anderson, Blelloch, Baweja, Acar, SPAA 2021)

Sources: [arXiv 2105.06712](https://arxiv.org/abs/2105.06712), [author PDF](https://danielanderson.net/papers/3409964.3461799.pdf), [ACM DOI](https://dl.acm.org/doi/10.1145/3409964.3461799), [cmuparlay/psac code](https://github.com/cmuparlay/psac).

- **Algorithm:** the trace is an **RSP tree** (Series-Parallel control tree: S-nodes ordered, P-nodes unordered, leaves = sequential strands, R-nodes = reads of *modifiables*). Change propagation re-executes affected reads **in SP order**: sequential deps respected, parallel siblings propagated concurrently — this replaces the sequential priority-queue-over-total-order (Dietz–Sleator order maintenance) used in classic SAC. Cost: `O(W_Δ + R_Δ·h)` work, `O(s·h)` span.
- **Reader sets** per modifiable: hand-rolled concurrent structures — inline single reader → lock-free linked list (small) → randomized BST keyed by hashed node addresses with **CAS insertion**; deletions mark dead + lazy cleanup; RSP subtree destruction deferred to a **post-propagation GC pile** to dodge races.
- **Numbers (32-core / 64-HT 4-socket AMD, 2.4 GHz):** initial-run tracking overhead **1.7×** (string hash) to **7.3×** (tree contraction); static parallel speedups 42–50× (spellcheck/hash/raytracer), 19–20× (contraction/filter); **work savings at k=1**: 819,600× (spellcheck), 770× (sequence), 1,180× (trees), 13,200× (filter); parallel speedup of propagation itself **22–39× for large batches** but "**little potential for parallelism for k=1 updates**"; from-scratch crossover at k≈10⁴ of n=10⁶ (contraction), k≈10⁵ of 10⁷ (filter), k≈10⁶/10⁵ of 10⁸ (hash, seq/parallel); memory 1–7× input (RSP tree: e.g. 20 MB list → 1.96 GB tree — granularity-tunable: 512-char leaves cut hash memory to 57.8 MB with 2.83 ms updates); reader-set scaling: update with 10⁶ readers of one modifiable only 4–5× slower than 1-reader case.
- **Direct lesson for signals:** parallel propagation pays only when the *frontier* is wide (batch updates, wide fan-out) and per-read work non-trivial; granularity control (chunking leaves) is the lever that makes both memory and update time acceptable.

## 5. The JS substrate: SharedArrayBuffer + Atomics — costs, latencies, prior art

**Measured microbenchmarks (this machine: Node v24.16.0, arm64/Apple Silicon, uncontended, hot 16-slot Int32Array):**

| op | ns/op |
|---|---|
| plain read (AB / SAB) | 0.49 / 0.53 |
| plain write (SAB) | 0.33 |
| `Atomics.load` | 4.59 |
| `Atomics.store` | 4.95 |
| `Atomics.add` | 4.89 |
| `Atomics.or` | 5.02 |
| `Atomics.compareExchange` | 6.33 |

≈**10× penalty per atomic op** (seq_cst; V8 doesn't yet emit relaxed ldar/stlr-free paths for JS atomics — [Mozilla bug 1077321](https://bugzilla.mozilla.org/show_bug.cgi?id=1077321) tracks acquire/release lowering). Cross-thread signaling round trips (Node `worker_threads`, same machine): **`Atomics.wait`/`notify` ping-pong 5.47 µs**, **`postMessage` ping-pong 7.90 µs** (tiny payload; browser postMessage is slower and payload-scaled). OS-level futex wait/notify baseline ≈10 µs ([computerBear: implementing atomic wait](https://blog.bearcats.nl/atomic-wait/)).

**Structured-clone scaling** ([Surma, "Is postMessage slow?"](https://surma.dev/things/is-postmessage-slow/), benchmarked Chrome/Firefox/Safari on 2018 MBP, Pixel 3XL, Nokia 2): transfer time ≈ proportional to `JSON.stringify` size above ~100 KiB; rule of thumb — **≤100 KiB fits a 100 ms interaction budget even on the slowest phones; only ≤10 KiB is safe inside a 16 ms frame budget**; PROXX's fix was **patchsets** (send diffs, chunked to frame budget) — i.e., exactly "ship dirty-flags, not state". `ArrayBuffer` *transfer* is near-constant-time regardless of size; manual JSON is not faster than structured clone.

**Platform rules & pitfalls:**
- `Atomics.wait` **throws on the main thread**; use [`Atomics.waitAsync`](https://v8.dev/features/atomics) (returns `{async, value|promise}`, promise resolves `'ok'|'timed-out'`) — workers can block, main thread must await.
- SAB requires **cross-origin isolation** (`Cross-Origin-Opener-Policy: same-origin` + `Cross-Origin-Embedder-Policy: require-corp|credentialless`) — [MDN SharedArrayBuffer](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/SharedArrayBuffer); Safari 15.2+/Chrome 68+/Firefox 79+ per [Partytown](https://partytown.qwik.dev/atomics).
- **Only numbers can be shared.** No shared JS object references, no atomics on object fields. Aligned integer typed-array accesses are tear-free even non-atomically (JS memory model); `Float64Array` accesses may tear; `Atomics.wait` works only on `Int32Array`/`BigInt64Array` views of SAB. Values (closures, objects, strings) must stay thread-local or be encoded (the marshalling cost Surma flags for the wasm/flatbuffer route).
- ES2024 **growable** SAB (`{maxByteLength}`) removes the realloc-and-rebroadcast problem for dynamic graphs.
- Practical patterns in the wild: atomic work-queues + version-stamped frames ([dev.to overview](https://dev.to/rigalpatel001/high-performance-javascript-simplified-web-workers-sharedarraybuffer-and-atomics-3ig1), [Giammarchi](https://webreflection.medium.com/about-sharedarraybuffer-atomics-87f97ddfc098), [blogtitle SAB guide](https://blogtitle.github.io/using-javascript-sharedarraybuffers-and-atomics/), [cross-tab sync](https://dev.to/hexshift/real-time-cross-tab-sync-with-sharedarraybuffer-and-atomics-no-server-no-broadcastchannel-2emj)); **[Partytown's Atomics mode](https://partytown.qwik.dev/atomics)** is the flagship production use — synchronous worker→main API proxying, claimed **10× faster** than its service-worker fallback.
- **No existing JS reactive/signal graph lives in SAB** as far as multiple searches can find — this would be novel territory.

## 6. Synthesis: what alien-signals could exploit

Grounding in the actual code (`upstream-alien-signals/src/system.ts`): nodes carry `flags: ReactiveFlags {Mutable, Watching, RecursedCheck, Recursed, Dirty=16, Pending=32}` plus doubly-linked `deps/subs` Link lists with per-link `version`; phases are `propagate` (push: walk subs, OR in Dirty/Pending, collect effects), `checkDirty` (pull: recursively verify Pending nodes, `update()` if needed), `shallowPropagate` (promote Pending→Dirty when a dep actually changed).

**(a) What parallelizes, per the literature:**
1. **Push-mark phase** is a monotone flag-OR graph traversal — embarrassingly parallelizable with `Atomics.or` on a flags word + frontier work-queue (classic parallel BFS). But at ~5 ns/atomic and alien-signals marking throughput of tens of ns/node single-threaded, a *single* wave must touch ~10⁴+ nodes before worker fan-out (≥5.5 µs wakeup each) breaks even. PSAC's k=1 result and FrameSweep's 160 µs/update threshold both say: **don't parallelize one small wave.**
2. **Pull/recompute phase** cannot cross threads generically: computed values are JS closures/objects, which are unshareable. Recompute parallelism therefore requires **graph partitioning by worker ownership** — each worker owns the values+closures of its partition; only topology/flags/versions live in the SAB. Cross-partition edges become: atomic flag-OR (mark) + `Atomics.notify`/`waitAsync` (wake owner to pull). This is FrameSweep's pipeline parallelism at partition granularity, with salsa's claim-CAS ensuring one owner recomputes each node.
3. **Independent transactions** (different input roots, disjoint or pipelined regions) are the real concurrency win per FrameSweep — e.g. main thread UI signals + worker-owned data-processing subgraph, with glitch-freedom per-partition and only eventual consistency across the seam (leptos' pragmatic position), unless you pay for a FrameSweep-style framing pass (placeholder counter word per node) across the seam.

**(b) Concrete SAB layout (SoA), if pursued:**
- `flags: Int32Array` — one word/node; bits 0–5 = current ReactiveFlags; add CLAIMED bit (salsa SyncTable-as-bit, CAS to claim recompute; `anyone_waiting` bit to skip notify); mark phase uses `Atomics.or`, returns old value so the first marker enqueues the node (dedup for free — exactly what `RecursedCheck/Recursed` do today, but race-safe).
- `version: Uint32Array` (per node) + global `revision: Uint32Array[durabilityTiers]` (salsa): cross-thread read validation = one atomic load; equal-value cut = don't bump version (salsa backdating ≈ existing `shallowPropagate` semantics).
- **Seqlock per shared scalar value**: writer `Atomics.add(seq,1)` (odd=writing), write payload words, `Atomics.add(seq,1)`; reader reads seq/payload/seq, retries if odd/changed — the only way to share actual values without locks, works only for numeric/fixed-width payloads (Float64 payloads need the seqlock because they may tear).
- Topology: CSR-style edge arrays (`subsOffset/subsList`, `depsOffset/depsList` as Int32 node indices) for the *shared, read-mostly* view; rebuild/patch under an epoch counter (double-buffer + epoch flip, readers check epoch — the cheap alternative to FrameSweep retrofitting; dynamic edges are the hardest part, and FrameSweep is the proof of how expensive full correctness is).
- Growable SAB for arena growth; free-list word for node-id recycling.

**(c) What I'd actually recommend exploiting first** (cost/benefit, given alien-signals' µs-scale updates):
1. **Typed-array SoA layout on its own (no threads)** — the flags/version/CSR layout above is worth benchmarking purely for cache locality and allocation wins; SAB-compat then comes almost free.
2. **Worker-owned subgraphs with SAB dirty-bridging**: replace postMessage state floods (10 KiB/frame ceiling) with atomic flag words + version counters + `waitAsync` wakeups (5–8 µs), values pulled lazily by the owning thread. This is Partytown's proven pattern applied to signals.
3. **Salsa-style cancellation** for long worker recomputes: one shared `AtomicBool`-equivalent word checked in `update()` loops; main-thread write bumps revision & flag; worker unwinds (throw) and re-pulls.
4. Skip full FrameSweep unless cross-thread strict consistency becomes a requirement — its 20–25% single-thread tax and 160 µs/update scaling threshold are worse than alien-signals' entire update budget.
