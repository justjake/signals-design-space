# Academic FRP glitch-freedom & update-scheduling: transferable algorithms for alien-signals

Scope: FrTime, Flapjax, the FrTime "Lowering" paper, Scala.React, Elm, REScala/SID-UP, and the Bainomugisha CSUR survey. All claims below come from reading the actual papers (PDFs fetched and read page-by-page), not abstracts. Where relevant I compare against alien-signals' actual scheme (read from `/Users/jitl/src/alien-signals-opt/upstream-alien-signals/src/system.ts`): `ReactiveFlags.Dirty`/`Pending` push-marking via `propagate`, pull-validation via recursive `checkDirty`, doubly-linked `Link` records with per-link `version`, and dependency re-linking during evaluation.

## Terminology anchor: the three ordering disciplines

The literature contains exactly three ways to get glitch freedom in a push system, and every surveyed system is one of them or a hybrid:

1. **Height/rank priority queue (topological scheduling)** — every node carries an integer rank > all producers' ranks; a priority queue processes dirty nodes in ascending rank. (FrTime, Flapjax, Scala.React, REScala's local engine, Scala.Rx.)
2. **Synchronous full-graph message waves** — no ranks; every node waits for exactly one message (`Change`/`NoChange`) per input per event; order is enforced by counting, not sorting. (Elm; SID-UP is a refinement that skips unaffected regions using source-ID sets.)
3. **Two-phase mark/pull ("mark-and-sweep")** — push a cheap "possibly dirty" mark down the graph, then validate lazily in dependency order by pulling. **This is alien-signals' family** (Dirty = definitely changed, Pending = possibly changed, `checkDirty` = the pull/sweep). Notably, the SID-UP paper states: *"mark-and-sweep algorithms generally outperform those based on topological sorting"*, citing the [Garnet/Amulet one-way dataflow constraints retrospective (Vander Zanden et al., TOPLAS 2001)](https://dl.acm.org/doi/10.1145/504709.504715) — a 20-year-old validation of the design alien-signals already uses, and a paper worth mining independently.

## 1. FrTime — height-keyed priority queue and the true cost of dynamic rank maintenance

Source: [Cooper & Krishnamurthi, "Embedding Dynamic Dataflow in a Call-by-Value Language" (ESOP 2006)](https://cs.brown.edu/~sk/Publications/Papers/Published/ck-frtime/paper.pdf); deeper implementation detail in [Cooper's PhD thesis](https://cs.brown.edu/people/ghcooper/thesis.pdf).

**Core algorithm.** Every signal stores an integer **height**. `make-signal` receives all producers, so height assignment at construction is trivial: `max(producer heights) + 1`. The engine runs a **priority queue processing nodes in order of increasing height** ("a slightly modified breadth-first search"). Since each signal is strictly higher than everything it depends on, no node evaluates before its producers — glitch-free by construction, and each node evaluates at most once per turn.

**Cutoff.** On recomputation the engine compares new vs. old value with `eq?`; if equal, dependents are not scheduled. (Same role as alien-signals' `update()` returning false + `shallowPropagate`.)

**Edges & GC.** Producers hold *reverse references* to dependents (push direction); these are **weak references** so unreachable signals get collected — but the paper admits dead signals may keep recomputing until GC runs, so FrTime adds explicit recursive deletion for dynamic branches (below). alien-signals' explicit `unlink` + `unwatched` callback is strictly cleaner here.

**Dynamic reconfiguration (§3.3) — the expensive part.** Time-varying conditionals/applications are implemented by a `dyn`/`fwd` node pair: the `dyn` node watches the trigger (e.g. an `if` condition), and on each trigger change constructs a whole new branch subgraph, which a permanent `fwd` node forwards from. The hazards, and FrTime's solutions, are exactly the costs alien-signals avoids by not having ranks:

- New branch nodes are created **uninitialized** (no values computed at construction) and simply *enqueued*; the propagation pass initializes them "after reaching the proper height," because the surrounding graph is mid-update and partially stale.
- "The new fragment's height may exceed that of the old one. To prevent glitches, the engine needs to **adjust height assignments to reflect the new graph topology** before performing any more updates. It must also **notify the priority queue of any changes in heights of signals that are already enqueued**" — i.e., dynamic edges force rank repair *plus* re-keying of in-flight queue entries.
- Deletion of the dead branch: rather than scrubbing the priority queue, FrTime **replaces each deleted signal's update procedure with a no-op** ("more efficient than removing the deleted signals from the priority queue"). The deletion set is "signals created within the branch's dynamic extent" — tracked during construction, including nested `dyn`s, deleted iteratively.
- Cycles: every cycle must pass through `delay`; **delayed signals are assigned height 0** because delay routes its value through the external event queue (no direct edge), which is what keeps the graph a DAG.

**Cost profile.** Per-update: PQ operations (log n or bucketed) + closure invocation; the Lowering paper enumerates the per-node update as five steps: "1) extracting the node from a priority queue, 2) retrieving the current value of its input signals, 3) invoking a closure to produce an updated value, 4) storing the new value in the signal object, and 5) iterating through a list of dependent signals and enqueueing them." Each signal object costs "nearly one hundred bytes" of heap. Contrast with alien-signals: no queue, no heights, dynamic dependency changes are O(1) `link`/`unlink` on the doubly-linked lists with zero ordering metadata to repair. FrTime's own conclusion concedes "significant overhead involved in moving from Scheme's top-down, stack-based execution model to FrTime's push-driven, queue-based update algorithm."

## 2. Flapjax (OOPSLA 2009) — rank PQ in JavaScript, trampolining, and runtime node compaction

Source: [Meyerovich, Guha, Baskin, Cooper, Greenberg, Bromfield, Krishnamurthi, "Flapjax: A Programming Language for Ajax Applications"](https://cs.brown.edu/people/sk/Publications/Papers/Published/mgbcgbk-flapjax/paper.pdf); backstory in the [SIGPLAN blog "Flapjax: From System to Paper"](https://blog.sigplan.org/2019/12/03/flapjax-from-system-to-paper/).

**Node representation (§3.2).** An event-stream node is `{sources: listof(EventStream), sinks: listof(EventStream), update: a -> (b or StopValue)}`. Constructors push the new node onto each source's `sinks` (Fig. 11). If `update` returns the `StopValue` sentinel, propagation stops at that node — `filterE` and behavior-equality cutoff are both expressed this way. Behaviors are event streams + current value; the behavior update function returns `StopValue` when the new value equals the current one.

**Propagation (§3.4).** Verbatim: "the node data structure tracks its partial-order **rank** in the dataflow graph. The Flapjax evaluator calls nodes' update functions in topological order. Instead of propagating values immediately, the evaluator **inserts them into a priority queue in topological order**." The recursive propagation model is "implemented with **trampolining**, to prevent stack overflow" — directly relevant to alien-signals' non-recursive `propagate`/`checkDirty` loops with explicit `Stack<T>`; Flapjax hit the same JS-engine constraint in 2007.

**Cycles.** "Flapjax expects that every cycle is broken by at least one delay (`delayE`) … This restores the delay-free sub-graph to a partial order."

**Higher-order streams and GC (§3.5).** `switchE` rewires `sources`/`sinks` at runtime (Fig. 12: remove `prevE` from `outE.sources`, remove `outE` from `prevE.sinks`). Because DOM listeners keep old inner streams alive, Flapjax adds an **`isDetached` flag computed during propagation**: a stream is detached iff all its sinks are detached; `$E` callbacks check it and call `removeEventListener` on themselves. This is a push-time reachability computation — alien-signals gets the equivalent for free via `unwatched`.

**Performance discussion (§4).** "The most significant cost in Flapjax is from **scheduling in the dataflow graph**. This breaks what might have been one large call-by-value evaluation into several small call-by-value fragments interspersed by dataflow graph manipulation and traversal. We have used several (sound) heuristics to **eliminate constants and to compact chains of nodes into single nodes, inspired by the lowering work of Burchett, et al.**" — i.e., Flapjax already did *runtime* graph compression (constant folding + linear-chain fusion) inside a JS signals runtime in 2009. No benchmark numbers are given ("considerable opportunity for performance improvement" is admitted); third-party users reported the implementation "could be faster."

## 3. Lowering (PEPM 2007) — node fusion / graph compression, the optimization signals libraries never adopted

Source: [Burchett, Cooper, Krishnamurthi, "Lowering: A Static Optimization Technique for Transparent Functional Reactivity"](https://cs.brown.edu/people/sk/Publications/Papers/Published/bck-lowering-opt-trans-frp/paper.pdf) (also [Burchett's slides](https://cs.brown.edu/media/filer_public/6a/53/6a530047-3b59-4f5e-9bf4-1ab07323dc9f/burchett.pdf)).

**Problem.** Transparent lifting creates one dataflow node per primitive application: `distance(x1,y1,x2,y2) = sqrt(sqr(x1-x2) + sqr(y1-y2))` allocates **six heap signal nodes** (~100 bytes each), and every input change pays queue+closure+store+enqueue per node.

**Mechanism.** A new form `dip`: `(dip (x…) e) ≡ ((lift (λ (x…) e)) x…)` — one dataflow node depending on all free variables of `e`, whose body runs as plain call-by-value code. A bottom-up static rewriting fuses dipped subexpressions:
- Literals: `⊢ c ⤳ (dip () c)`; identifiers: `⊢ id ⤳ (dip (id) id)`.
- Application: if the function has a known **lower counterpart** (unlifted version), union the argument dips' dependency lists and replace with the lower function: `(f̂ e₁…) ⤳ (dip (x⃗₁ ∪ …) (f e₁'…))`.
- **Conditionals**: `(if c t f) ⤳ (dip (x⃗_c ∪ x⃗_t ∪ x⃗_f) (if c' t' f'))` — legal but the fused node depends on the **union of both branches' dependencies**, sacrificing the dynamic-dependency pruning that a graph gives you.
- **Lambdas**: only the *body* is dipped (dipping the whole λ is unsafe — the lowered ops would see signals); successful body-lowering yields a lower counterpart for interprocedural fusion (name-mangling convention lets it work across modules).
- **Not lowerable**: higher-order functions (`map`'s function argument may itself produce signals), functions with history (`delay`), and allocating data constructors (FrTime uses in-place-mutating `cons`/`append` variants to avoid reallocation; blind fusion would reintroduce churn). It is always safe to stop fusing anywhere — lowering is opportunistic.

**Benchmarks (Table 1)** — the strongest fusion numbers in the literature:

| Benchmark | Size (exprs) | Mem orig→opt | Shrinkage | Run orig→opt | **Speedup** |
|---|---|---|---|---|---|
| Count (microbench) | 7 | 204.7→0.2 MB | 971× | 4.8→<0.1 s | **16,000×** |
| Needles | 62 | 581.4→240.5 MB | 2.4× | 5.6→2.0 s | **2.8×** |
| Spreadsheet | 2,663 | 34.8→50.9 MB | 0.7× (worse) | 19.3→20.5 s | **0.94× (slower)** |
| TexPict (Slideshow) | 13,022 | 170.7→119.4 MB | 1.4× | 273.4→3.5 s | **78.1×** |

**The pathological case (§3.9) — critical for alien-signals.** `(expensive-operation (quotient milliseconds 10000))`: unfused, the `quotient` node's equality cutoff stops propagation ~9,999 times out of 10,000; fused, the whole expression "must recompute *in its entirety* each time milliseconds changes." **Fusion deletes intermediate cutoff points.** That is exactly why the Spreadsheet benchmark regressed. Any fusion in alien-signals must preserve cutoffs where value-change ratios differ across a chain (or only fuse when the intermediate has exactly one subscriber and no equality-filtering effect).

**Transfer assessment.** No modern TS signals library (alien-signals, Reactively, Preact Signals, MobX) fuses nodes. Realistic variants for alien-signals: (a) **linear-chain collapse at runtime** — a computed whose only subscriber is another computed and which is not `Watching` can be inlined into its subscriber (Flapjax did precisely this dynamically); (b) **constant elimination** — links to nodes that can never change again (no `Mutable` upstream) can be unlinked wholesale; (c) the paper's future-work note applies directly: "For languages that support runtime code generation, it would be possible to explicitly build the dataflow graph first, and then collapse nodes into call-by-value subexpressions … whether or not they contained un-lowerable subexpressions." Related precedent cited: Yampa's runtime composition-collapsing of pure signal functions, and Nilsson's GADT-based constant-propagation/identity-elimination (Elm's thesis reports "Arrowized FRP was able to achieve a speed-up of two orders of magnitude by collapsing all of their signal functions … into a single monolithic node").

## 4. Scala.React — level PQ with level-mismatch abort/retry, and lazy nodes ≈ alien-signals

Source: [Maier & Odersky, "Deprecating the Observer Pattern with Scala.React", EPFL-REPORT-176887 (2012)](https://infoscience.epfl.ch/entities/publication/e606e195-5e7a-4ba2-9d30-3205bb42201a) ([PDF](https://infoscience.epfl.ch/bitstreams/edac8552-b524-41c2-a4af-23abf42a0821/download)); earlier version ["Deprecating the Observer Pattern" (2010)](https://infoscience.epfl.ch/record/148043).

**Turns and propagation (§7, §7.1).** Discrete **propagation turns**; a thread-safe scheduler admits external writes for the *next* turn. "When a propagation turn starts, the propagator puts all nodes that have been invalidated since the last turn into a **priority queue sorted according to the topological order, briefly *level***, of the nodes. The propagator dequeues the node on the lowest level and validates it, potentially … putting its dependent nodes, which are on greater levels, on the queue … always keeping track of the current level." Monotonic level progression ⇒ glitch freedom.

**Strict vs. lazy nodes (§7.2) — the direct ancestor of Dirty/Pending.** Strict nodes validate on notification and can cut off propagation when they don't change (`filter`, `Strict{…}` signals, stateful `count`/`scan`). **Lazy nodes** (from `map`, `merge`, `Lazy{…}`) "are notified by their dependencies eagerly, but **validated only when queried** … a lazy node simply sets an internal bit that indicates that it has been invalidated if not set yet. The node is not inserted into the queue for strict evaluation, but always notifies its dependents when it gets invalidated, which happens **at most once per turn**. Since they evade the propagation queue, lazy nodes have a performance advantage… Their disadvantage … they have to notify their dependents regardless of whether they will actually emit." That is alien-signals' `Pending` mark-push + pull-validate, embedded inside a level-queue system; Maier's benchmarks confirm the mark/pull side is the fast path ("strict signals have a fairly high overhead … they evaluate the paths through the graph piecewise, instead of in one large call chain").

**Opaque nodes & dependency discovery (§7.3).** Signal expressions `Signal{ a() + b() }` are **opaque**: dependencies are discovered by evaluation using "a stack of dependent reactives" (identical to alien-signals' active-subscriber tracking). Cleanup is lazy and *over-approximating*: "when a node notifies its dependents of a change, it removes all dependents that are flagged as opaque" — deps re-subscribe on next evaluation; until then a node may be notified by a dependency it no longer reads ("trade a bit of redundant computation for the benefits of a single general implementation"). alien-signals is more precise (per-evaluation `depsTail` truncation + version-stamped links), at the cost of link bookkeeping per evaluation. §7.3 also contains an explicit comparison to Lowering: runtime dependent-stacks handle "runtime dependencies and higher-order programming without further adaptation," which static lowering cannot (e.g. `sum(xs: List[Signal[Int]])`).

**Level mismatches (§7.6) — dynamic rank repair, priced.** When an opaque node `n` mid-evaluation reads a node whose level ≥ the current propagation level, the read **"throws a level mismatch exception containing a reference to itself, which is caught … in the main propagation loop. The propagator then hoists `n` by first changing its level to a level above the node which threw the exception, reinserting `n` into the propagation queue … and then transitively hoisting all of `n`'s dependents."** Consequences: expression signals get **evaluated redundantly from the beginning** after an abort ("should therefore never perform external side-effects"); CPS-based flow reactives capture their continuation before each read so they *resume* at the abort point. Routers sit one level above the max of their dependencies; reactors are pinned at maximum level so they can never abort. This abort-hoist-retry machinery — plus transitive dependent re-leveling — is the concrete price of keeping ranks consistent under dynamic reads; alien-signals' `checkDirty` recursion needs none of it.

**Memory layout note (§7.7.1).** All forward (node→dependent) references are weak; "every node that can have dependencies stores a reference to a **single weak reference object to itself**" (never a HashMap — "a HashMap creates additional garbage").

**Benchmarks (§9, JVM i7 2.4GHz, vs. Flapjax on Chrome 15/V8).** Chain of 100 nodes / fan-out 100, 1000 events per iteration: plain observers win everywhere (313k iterations/s JVM). Scala.React is "about factor **3 faster** [than Flapjax] for event chains, and from factor **5 to about factor 7** for signal chains … For fans, Scala.React is relatively faster by factor **4 to 10**." Lazy signals ≈ event-combinator speed; strict signals notably slower.

## 5. Elm (Czaplicki & Chong, PLDI 2013) — synchronous message waves, NoChange memoization, pipelining

Sources: [PLDI 2013 paper "Asynchronous Functional Reactive Programming for GUIs"](https://dl.acm.org/doi/10.1145/2491956.2462161) ([PDF](https://people.seas.harvard.edu/~chong/pubs/pldi13-elm.pdf)); mechanism extracted from [Czaplicki's thesis "Elm: Concurrent FRP for Functional GUIs"](https://elm-lang.org/assets/papers/concurrent-frp.pdf) (ch. 4, which the paper formalizes).

**Static DAG, no ordering metadata.** Elm's type system bans higher-order signals, so the signal graph is **fixed at compile time**. Translation to Concurrent ML: "Each node is a concurrent thread, and each edge is a [FIFO] channel."

**The wave protocol.** A **global event dispatcher** receives all external events through a queued mailbox and "notifies **all** input nodes [with the event ID], even if the update does not change the node." Then: "Multi-input nodes must receive a value from each input edge before computing a result"; unaffected inputs forward `NoChange prev`. A lift node "waits to receive a message from every input channel. It then checks to see if any of the inputs have changed. If so, the new output value is computed and a change is reported. Otherwise, the node sends a 'no change' message with the most recent output value" (Fig. 6 gives the full CML translation: `datatype 'a event = NoChange 'a | Change 'a`). Glitch freedom and event ordering fall out of FIFO edges + one-message-per-input-per-event — **no ranks, no queue, no marks**. `NoChange` doubles as memoization ("These messages tell nodes to send along the latest computed value rather than computing it again").

**Cost.** Every node and every edge processes a message for **every** global event — O(V+E) per event even if nothing relevant changed. SID-UP's authors: "processing every update turn at every node implies a lot of wasted computational resources." alien-signals touches only the transitively-reachable-from-changed region (and pulls only what's observed), which strictly dominates in a single-threaded setting.

**What Elm buys instead.** (a) **Pipelining** — multiple turns in flight as sequential wave fronts (each edge buffers pulses per turn); (b) **`async`** — a local graph rewrite that cuts a subgraph out and re-injects its outputs as new *input* events via the dispatcher, so slow computations don't block unrelated updates ("Event order is maintained *within* each subgraph, but not *between* them"). Caveat noted by Drechsler et al.: Elm's pipelining is "inherently incompatible with dynamic dependencies" — edges created mid-turn have no buffered history, which is why no dynamic-graph JS library can use it directly. The transferable concept for alien-signals is scheduling-level, not graph-level: `async` ≈ splitting effect scopes so expensive `Watching` subtrees run in a later microtask without breaking per-subgraph ordering.

## 6. REScala / SID-UP (OOPSLA 2014) — glitch freedom from reachable-source sets, no heights, no central queue

Source: [Drechsler, Salvaneschi, Mogk, Mezini, "Distributed REScala: An Update Algorithm for Distributed Reactive Programming"](https://dl.acm.org/doi/10.1145/2660193.2660240) ([PDF](https://programming-group.com/assets/pdf/papers/2014_Distributed_REScala_An_Update_Algorithm_for_Distributed_Reactive_Programming.pdf)). (REScala's plain local engine is level-based like Scala.React — [Salvaneschi et al., AOSD 2014](https://dl.acm.org/doi/10.1145/2577080.2577083); the thread-safe successor is FrameSweep, ["Thread-Safe Reactive Programming", OOPSLA 2018](https://dl.acm.org/doi/10.1145/3276477).)

**Data per node.** Each source input `i` has a unique ID. Every node `n` stores `sₙ` = the set of source IDs from which `n` is transitively reachable, computed at node creation as the union of its dependencies' sets. Per turn, each node has a pulse state `pₙ ∈ {pending, unchanged, changed}` (reset to `pending` each turn).

**Turn protocol.** Admission phase: collect `t = {ids of sources changed this turn}`; only changed sources pulse `changed`, and `t` rides on every pulse. Propagation phase, on node `d` receiving a pulse while `p_d = pending`: for each incoming dependency `dd`, compute `x_dd = t ∩ s_dd`. If some `dd` has `p_dd = pending ∧ x_dd ≠ ∅` — i.e., *that input can still be affected this turn but hasn't reported* — `d` waits. Once every potentially-affected input has pulsed, `d` reevaluates iff some input pulsed `changed`, else immediately pulses `unchanged` without reevaluating. Each node evaluates **at most once per turn**; proof is by induction over "all predecessors transitively reachable from changed sources have updated glitch-freely." So: glitch freedom with **no heights, no priority queue, no central coordinator during propagation** — ordering knowledge is factored into per-node source sets + pulse counting.

**Dynamic dependencies (§3.2).** Source-ID sets are themselves maintained *by the same propagation mechanism*: a second per-node `changed/unchanged` flag tracks set updates; "the node's source identifier set is recalculated when any dependency's source identifier set changed," and a node's single pulse per turn is emitted only after both value and set recalculation. Dynamic nodes that swap dependencies mid-wait "simply re-perform the same waiting checks on these new dependencies." Cost under heavy graph churn: set unions/intersections on every affected node (the authors admit SID-UP "with its comparatively very complex set operations has the biggest overhead per node"), and set sizes grow with input count — their sensitivity experiment with ~150 extra sources showed SID-UP "slightly worse" while other algorithms were unaffected.

**Complexity comparison (§4.3) — useful mental model.** With `u` = nodes actually updated, `c` = |CC| nodes transitively reachable from changed sources, `n` = all nodes: messages are `M_Scala.React ~ u² + u` (queue/coordinator round-trips; ~`2u` for sparse graphs), `M_SID-UP ~ c²` (~`c` sparse), `M_ELM ~ n²` (~`n` sparse); `u ≤ c ≤ n`. Turn latency: level-ordered engines serialize independent branches by level (`Σ max(tᵢ)`), while SID-UP/ELM let branches progress independently (`max(Σ tᵢ)`). In alien-signals terms: `propagate` already touches ~CC once (flag marking), `checkDirty` touches ~the pulled slice of CC; there is no coordinator overhead at all — the paper's own numbers imply a sequential in-process library loses nothing by not adopting SID-UP, whose wins are remote-message counts and intra-turn parallelism. The genuinely transferable ideas: (a) **source-set intersection as a cheap "can this Pending node possibly be affected?" filter** — a bloom-filter/bitmask of source IDs per node could let `checkDirty` skip whole subtrees when the changed-source set doesn't intersect (interesting for very wide graphs with localized writes); (b) the per-turn tri-state `pending/unchanged/changed` is a cleaner formalization of Pending-resolution than re-deriving it from flags each time.

**Empirical.** On simulated-latency distributed topologies (chain/regular/fan modules, degree distributions measured from 20 real reactive apps — Fig. 11: median in-degree 1, out-degree ≤ 2, max 22), SID-UP completes turns fastest as latency grows; locally the case study runs ~10% slower than hand-written observers. Also note their remark that topology optimization "The most prominent technique here is **Lowering** [Burchett]… orthogonal to the propagation algorithm; we expect such optimizations to benefit SID-UP the most" — the fusion + propagation-algorithm interaction is compositional.

## 7. Bainomugisha et al., "A Survey on Reactive Programming" (ACM CSUR 2013) — where alien-signals sits

Source: [ACM CSUR 45(4)](https://dl.acm.org/doi/10.1145/2501654.2501666) ([VUB tech report PDF](https://soft.vub.ac.be/Publications/2012/vub-soft-tr-12-13.pdf)).

Taxonomy axes: **basic abstractions** (behaviours/events vs. cells), **evaluation model** (push / pull / both), **lifting** (implicit / explicit / manual), **multidirectionality**, **glitch avoidance**, **distribution**. Table I classifies 15 languages: pull = Fran, Yampa; push = FrTime, Flapjax, Scala.React, Frappé, AmbientTalk/R, Trellis, SuperGlue, Coherence, .NET Rx; both = NewFran (Elliott's [push-pull FRP 2009](http://conal.net/papers/push-pull-frp/)), Lamport Cells. Glitch avoidance: "Most reactive programming languages eliminate glitches by arranging expressions in a topologically sorted graph" (citing FrTime/Flapjax/Scala.React); exceptions: Radul/Sussman propagators (none), Trellis (`Y*`, needs program structuring), .NET Rx (`N?` — no glitch-freedom semantics). The survey flags distributed glitch freedom as the open problem (answered a year later by SID-UP). Notably the survey's hybrid push-pull category ("reap the benefits of the push-based model (efficiency and low latency) and those of the pull-based model (flexibility of pulling values based on demand)", demonstrated in Lula and NewFran) is precisely alien-signals' quadrant; the academic lineage stopped at coarse push-pull hybrids and never described the modern flag-bit + version-counter refinement — there is no academic prior for `Link.version`-style re-validation, whose closest relative is Scala.React's per-turn "validity" and Adaptive Functional Programming's timestamps (Acar et al., which Maier dismisses as "much more heavyweight … based on time stamps and a selective replay mechanism").

## Synthesis: ordering discipline × dynamic-graph maintenance cost

| System | Ordering mechanism | Cost under dynamic dependency change | vs. alien-signals Dirty/Pending |
|---|---|---|---|
| FrTime | height = max(prod)+1; PQ by ascending height; `eq?` cutoff | Rebuild branch uninitialized + enqueue; **repair heights & re-key already-enqueued PQ entries**; delete via no-op update fns; delay ⇒ height 0 | alien-signals: zero ordering metadata to repair; O(1) link/unlink |
| Flapjax | same rank PQ; `StopValue` cutoff; trampolined | `switchE` rewires sources/sinks; `isDetached` recomputed during propagation for GC | trampolining anticipates alien-signals' explicit stacks; runtime chain-compaction is the idea worth stealing |
| Scala.React | level PQ per turn; strict nodes cut off | **Level-mismatch exception → hoist level → re-enqueue → transitively hoist dependents → redundant re-eval** (or CPS resume); levels only grow | its *lazy nodes* (invalidate-bit + notify-once + validate-on-query) are literally Pending; alien-signals generalizes the fast path to the whole system |
| Elm | none — static DAG + FIFO edges + one msg/input/event (`Change`/`NoChange`) | none possible (static graph is the precondition; pipelining breaks under dynamic edges) | O(V+E) per event vs. alien-signals' O(affected); Elm buys pipelining/parallelism instead |
| SID-UP | per-node reachable-source-ID sets; wait while ∃ input pending ∧ t∩s ≠ ∅; tri-state pulses | source sets recomputed via the same propagation wave (second changed-flag); set ops are the dominant per-node overhead | Pending-resolution by counting pulses (push-complete) instead of pulling; source-set intersection = potential skip-filter for `checkDirty` |

## Concrete candidate ideas for alien-signals extracted from this literature

1. **Runtime linear-chain fusion (Lowering / Flapjax §4).** Collapse computed chains where the intermediate has exactly one subscriber, is unwatched, and (to respect the §3.9 pathology) either has no equality cutoff benefit or its value changes whenever its input does. Expected win by analogy: Flapjax deemed scheduling *the* dominant cost; Lowering got 2.8–78× on real programs. This is the one major technique no TS signals library has adopted.
2. **Constant/dead-region elimination (Lowering; Flapjax).** Nodes with no `Mutable` ancestor can be snapshotted and unlinked; FrTime/Flapjax did this soundly at graph-construction time.
3. **Source-ID bitmask skip-filter (SID-UP).** Per-node bitset (or bloom filter) of reachable source IDs; `propagate`/`checkDirty` can skip marking/pulling subtrees whose mask misses the writing source. Pays off only for wide graphs with localized writes; SID-UP's own data warns the set maintenance can dominate — a fixed-width bitmask (e.g. 64 sources hashed) bounds that cost.
4. **Don't adopt heights/levels.** Every height-based system pays rank repair under dynamic edges (FrTime's PQ re-keying; Scala.React's abort-hoist-retry with redundant re-evaluation). The mark-and-sweep family alien-signals is already in was found to "generally outperform" topological sorting (Vander Zanden TOPLAS 2001, endorsed by Drechsler 2014). The only thing levels buy over pull-validation is single-pass push-completion and easy layer-parallelism — irrelevant single-threaded.
5. **Elm's `async` as scheduler policy.** Partition effect subscribers into ordering domains; expensive watchers re-enter via the scheduler as new "input events" rather than running inside the turn. Preserves per-domain consistency without global blocking.
6. **Mine Vander Zanden et al. TOPLAS 2001** (Garnet/Amulet one-way dataflow constraints): the direct empirical ancestor of the mark-sweep-beats-topological claim, with lazy vs. eager evaluation data and cycle-handling ("once-around" DFS that stops at already-updated nodes — no delays needed), predating all FRP work.
