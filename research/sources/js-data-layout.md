# Data-Oriented Design in JavaScript: Research Report for a Typed-Array Rewrite of alien-signals

## 0. Executive summary (what matters most for alien-signals)

1. **The closest prior art to what you want to do is Fitzgerald/Osipov's `source-map` optimization**, documented in mraleph's ["Maybe you don't need Rust and WASM to speed up your JS"](https://mrale.ph/blog/2018/02/03/maybe-you-dont-need-rust-to-speed-up-your-js.html): hundreds of thousands of small `Mapping` objects (structurally identical to alien-signals' `Link` objects) were replaced with a single `Int32Array` where each record occupies 6 consecutive slots. Combined with monomorphization and cache removal, total speedup was **4.2x on V8 (4810ms → 1143.5ms)** and 4x on SpiderMonkey. This is the strongest empirical evidence that "millions of small graph-edge objects → flat Int32Array" pays off in JS.
2. **No production signals library currently uses typed-array/SoA storage** — Preact, alien-signals, and Vue 3.5+ use doubly-linked lists of `Node`/`Link` objects; [reactively](https://github.com/milomg/reactively) uses plain JS arrays of object references. A SoA/typed-array signals engine appears to be genuinely unexplored territory in JS (the nearest analogues are Rust's [leptos, which stores its whole reactive graph in a slotmap arena and hands users `Copy` index handles](https://github.com/leptos-rs/leptos/blob/main/ARCHITECTURE.md), and the JS ECS world).
3. **The JS ECS ecosystem (bitECS, wolf-ecs, piecs) has already solved the mechanical subproblems you need**: SoA typed-array field storage, sparse-set membership with swap-remove, index recycling with generation/version bits packed into a `Uint32`, and fixed-capacity preallocation. Benchmarks ([js-ecs-benchmarks](https://github.com/ddmills/js-ecs-benchmarks)) show typed-array ECSs beating object-based ones ~2x on iteration.
4. **Beware the two classic failure modes**: (a) values are polymorphic — signal *values* must stay in a plain `PACKED_ELEMENTS` side array; only the graph topology/flags/versions belong in typed arrays; (b) WASM is not a shortcut — Surma's [experiments](https://surma.dev/things/js-to-asc/) show allocation-heavy WASM (AssemblyScript with GC) ran **80x slower** than JS, and per-call JS↔WASM boundary overhead (~100ns) multiplied by per-access `signal.get()` calls would dominate.

## 1. Struct-of-arrays in JS: how the ECS libraries do it

### bitECS ([intro docs](https://github.com/NateTheGreatt/bitECS/blob/main/docs/Intro.md), [site](https://bitecs.dev/docs/introduction))
- Components are **SoA: one array per field, indexed by entity id** — `const Position = { x: new Float64Array(10000), y: new Float64Array(10000) }`. Docs recommend: typed-array SoA for threading + zero GC; plain-array SoA for minimal footprint; AoS objects only "so long as the shape is small and there are < 100k objects."
- **Entity id recycling** ([EntityIndex.ts](https://raw.githubusercontent.com/NateTheGreatt/bitECS/main/src/core/EntityIndex.ts)): a **dense/sparse array pair**. `dense` holds alive ids contiguously; `sparse[id]` gives the position in `dense`. Removal **swap-removes**: the target swaps with the last alive entry and `aliveCount--`. Allocation recycles automatically when `aliveCount < dense.length`, else `++maxId`. No explicit free-list structure needed.
- **Generation counters ("versioning")**: an id is a `Uint32` split into entity bits (low) and version bits (high): `entityBits = 32 - versionBits; entityMask = (1<<entityBits)-1; versionShift = entityBits`. With 8 version bits you get **16M entities × 256 recycles**; 12 bits → 1M × 4096. Staleness check = extract id, check `sparse[id] < aliveCount && dense[sparse[id]] === queriedId`. Caveat from the docs: versioned ids can't be used directly as typed-array indices (the high bits make them huge) — you must mask first.
- Queries can return a **`buffered: Uint32Array`** result instead of a JS array; hierarchy queries return entities in **topological order (parents before children)** with cached depth values — directly analogous to a depth-ordered recompute queue for signals.

### wolf-ecs ([repo](https://github.com/EnderShadow8/wolf-ecs))
- [`component.ts`](https://raw.githubusercontent.com/EnderShadow8/wolf-ecs/main/src/component.ts): each declared field type maps 1:1 to a TypedArray constructor (i8/u8/i16/u16/i32/u32/f32/f64/i64/u64) allocated **once at fixed `max` capacity**; `any` falls back to `new Array(max)`.
- [`sparseset.ts`](https://raw.githubusercontent.com/EnderShadow8/wolf-ecs/main/src/sparseset.ts): textbook sparse set — `has(x)` is `sparse[x] < packed.length && packed[sparse[x]] === x`; remove is pop-and-swap; all O(1), and iteration happens over the dense `packed` array.

### miniplex ([repo](https://github.com/hmans/miniplex)) — the deliberate counter-example
Entities are **plain JS objects, components are properties**; the author explicitly trades raw speed for DX ("making its usage feel as natural as possible in a JavaScript environment"). Useful as the control group: in [js-ecs-benchmarks](https://github.com/ddmills/js-ecs-benchmarks), miniplex iteration (velocity test) is **20ms vs 9–12ms** for the typed-array libs (piecs 9ms, wolf-ecs 11ms, bitECS 12ms), and add/remove is 66ms vs 32–52ms. So the SoA typed-array approach buys roughly **2x on iteration** even in well-written object code — a moderate but real ceiling raise.

### Javelin
The javelin blog ("[ECS in JS – Storage Mechanisms](https://javelin.hashnode.dev/ecs-in-js-storage-mechanisms)", "[A Closer Look at Component Storage](https://javelin.hashnode.dev/ecs-in-js-a-closer-look-at-component-storage)") is Cloudflare-gated, but its conclusions are reflected in the benchmark table above (javelin-ecs iterates at 18ms, ~2x slower than the typed-array leaders — it uses archetype tables of object components).

**Transferable pattern for alien-signals:** reactive nodes = "entities" (a `Uint32` index with optional generation bits); node fields (`flags`, `version`, `depsHead`, `subsHead`, `depth`, …) = SoA typed arrays; links = a second arena of parallel `Uint32Array`s (`dep`, `sub`, `prevSub`, `nextSub`, `nextDep`…) — i.e., you can keep the exact linked-list *algorithm* while changing only the *layout*, replacing 64-bit pointers with 32-bit indices.

## 2. V8 specifics you must design around

### Object cost model (why Link objects are expensive)
- Every JS object carries a header of **3 words: map (hidden class) pointer, properties pointer, elements pointer = 24 bytes on x64, 12 bytes with pointer compression**, before any fields ([A tour of V8: object representation](https://jayconrod.com/posts/52/a-tour-of-v8-object-representation), [JSObject structure analysis](https://medium.com/@bpmxmqd/v8-engine-jsobject-structure-analysis-and-memory-optimization-ideas-be30cfcdcd16)).
- [Pointer compression](https://v8.dev/blog/pointer-compression): pointers/tagged fields are 4 bytes within a 4GB "cage"; tagged values were measured as **~70% of the V8 heap**, and compression cut V8 heap size **up to 43%**. Smis are **31-bit** signed ints under compression.
- So one alien-signals `Link` (7 tagged fields: `dep, sub, prevSub, nextSub, prevDep, nextDep, version`-ish) ≈ **12B header + 7×4B = 40 bytes** heap + GC tracing of 6 pointers. The SoA equivalent is **24–28 raw bytes** in `Uint32Array`s with **zero GC-visible pointers and zero write barriers** (typed-array stores are raw numbers — the write barrier only fires when writing *heap pointers* into objects).
- [Slack tracking](https://v8.dev/blog/slack-tracking): constructors start with expected-props **+8 in-object slots**, and after **7 instances** the map is finalized and shrunk. Classes with many fields spill extra properties to the out-of-object backing store (extra indirection) — keep node classes ≤ ~10 fields if you keep objects at all, and always initialize every field in the constructor in the same order ([Fast properties in V8](https://v8.dev/blog/fast-properties)).

### Hidden classes, ICs, megamorphism
- [Fast properties](https://v8.dev/blog/fast-properties): objects with identical property sequences share a hidden class; inline caches key on it; dictionary-mode properties never benefit from ICs.
- [mraleph, "What's up with monomorphism"](https://mrale.ph/blog/2015/01/11/whats-up-with-monomorphism.html): IC states are monomorphic → polymorphic (**linear search, threshold ~4 shapes for property loads in V8**) → megamorphic (**global fixed-size hash table with collision overwrites — near-worst-case every time**). Crucially, polymorphic accesses also block redundancy elimination in TurboFan: after a monomorphic check the compiler *knows* the shape until the next side effect and can drop later checks; polymorphic decision trees can't be deduped.
- **Relevance:** if `Signal`, `Computed`, and `Effect` are distinct shapes and hot code does `node.flags` / `link.sub.flags` over mixed kinds, those sites are polymorphic (3-4 shapes) forever. Two fixes: (a) unify all node kinds into one hidden class (one class, unused fields null — this is what alien-signals mostly does already); (b) go SoA — `flags[id]` on a `Uint32Array` is always monomorphic no matter how many node kinds exist. (b) is strictly stronger.

### Elements kinds ([V8 blog: Elements kinds](https://v8.dev/blog/elements-kinds))
- 21 kinds; transitions are one-way down the lattice: `PACKED_SMI → PACKED_DOUBLE → PACKED_ELEMENTS`, and any `PACKED → HOLEY` transition is **permanent** ("once holey, always holey").
- `new Array(n)` creates a **holey** array; build side arrays with `push`, or accept holey-ness, or preallocate and densely fill (still holey if created with a length). Avoid `-0`, `NaN`, `Infinity` in what should be Smi arrays (they force DOUBLE).
- **Out-of-bounds reads poison the IC**: fixing `i <= arr.length` to `i < arr.length` gave a documented **6x** improvement. Never read `arr[-1]`-style sentinels.
- Plain `PACKED_SMI` arrays are often *as fast to read* as typed arrays (both are flat stores); typed arrays win on: guaranteed kind stability (no transition checks, no holes possible), no Smi range cliff, 2-8x smaller memory per element for narrow types, no GC tracing of elements, and `SharedArrayBuffer`/`Atomics` compatibility.

### Numbers-in-fields cliff ([The story of a V8 performance cliff in React](https://v8.dev/blog/react-cliff))
- Field *representations* also form a one-way lattice: Smi → Double → Tagged. A version counter stored as an object field that ever exceeds the **31-bit Smi range (±1,073,741,823 under pointer compression)** flips the field to Double representation (boxed `MutableHeapNumber` mutation or shape migration). React hit this with `performance.now()` timestamps in `FiberNode` and worked around it by initializing with `Number.NaN` to force Double from the start. In a `Uint32Array`, a version counter is always 4 raw bytes — no representation games. (alien-signals-style global version/epoch counters can plausibly exceed 2^30 in long-lived apps: ~12 days at 1000 updates/sec.)

## 3. Canonical writeups worth reading in full

- [mraleph — Maybe you don't need Rust and WASM to speed up your JS](https://mrale.ph/blog/2018/02/03/maybe-you-dont-need-rust-to-speed-up-your-js.html). Full technique list with numbers: (1) fixing arity mismatch to kill `ArgumentsAdaptorTrampoline` (**14%**, 4774→4123ms); (2) **monomorphizing a generic sort by cloning the sort function per comparator** via `new Function` (**25%**, 4268→3177ms) — directly applicable if alien-signals ever takes user comparators/equality fns; (3) deleting a memoization cache because `KeyedLoadIC_Megamorphic` dominated the profile (caching by string key was slower than recomputing); (4) bucket-sort + insertion-sort exploiting near-sortedness; (5) **packing all mappings into one `Int32Array`, 6 slots per record, objects becoming thin lazily-materialized wrappers holding a base index**; (6) replacing `charCodeAt` with `Uint8Array` reads. Net **4.2x**.
- [thomcc — High Performance JavaScript via Data-Oriented Design](https://www.thomcc.io/2015/09/06/high-performance-javascript.html): replace a list of JS objects with typed arrays + a manager; allocate **one ArrayBuffer and split it into field views** for locality; result: 80k simulated springs in <1ms/frame. Notes the weak spot: dynamic add/remove needs preallocation/ring-buffer strategies — exactly what the ECS sparse-set/free-list machinery solves.
- [Surma — Is WebAssembly magic performance pixie dust?](https://surma.dev/things/js-to-asc/) (details in §5).
- [Structurae](https://github.com/zandaqo/structurae) ([intro article](https://medium.com/@zandaqo/structurae-data-structures-for-high-performance-javascript-9b7da4c73f8)): off-the-shelf **typed-array graphs** — `AdjacencyList` "extends a TypedArray… storage = vertices + edges×2", adjacency matrices, `Pool` (free-slot tracking via a BitArray: `pool.get()`/`pool.free(i)` with O(1) bit-scan), grids, and `RecordArray`/View binary structs (C-like structs over `DataView`). Good reference implementations for edge storage and bitset-based free lists.

## 4. Linked lists, graphs, pools, and growth in typed arrays

### Patterns
- **Index-based free list**: store `nextFree` in the same `Uint32Array` slot that normally holds e.g. `nextDep`; `freeHead` is a scalar. Alloc = `if (freeHead !== NIL) { i = freeHead; freeHead = next[i]; } else i = count++`. Zero allocation, zero GC. (This is exactly how you'd port alien-signals' `Link` recycling; alien-signals already pools one link per dep-sub pair at the object level, per the same design in [Preact's Signal Boosting](https://preactjs.com/blog/signal-boosting/).)
- **Generational indices / slotmaps** ([guide](https://lucassardois.medium.com/generational-indices-guide-8e3c5f7fd594), [deno slotmap](https://deno.land/x/slotmap@0.1), bitECS versioning above): pair every slot with a generation counter; handles embed (index, generation); stale handles fail the generation check instead of aliasing a recycled slot. The Deno slotmap uses the elegant "odd generation = vacant" encoding. For signals this matters if user code can hold a `Signal` handle after `dispose()` — either use generation bits inside the packed `Uint32` handle (bitECS style) or keep a user-facing wrapper object per node (one object per *node* is fine; it's the per-*edge* objects that dominate).
- **Epoch/endIndex instead of clearing**: rather than truncating a deps array on re-track, keep a `depsCount`/epoch and overwrite in place, only trimming links past the final index — reactively does the object-array version of this: it walks new deps with an index pointer and only touches `observers` back-links where old and new diverge ([reactively core.ts](https://raw.githubusercontent.com/milomg/reactively/main/packages/core/src/core.ts)).

### Growing storage
- Manual strategy (works everywhere, predictable): geometric doubling — allocate `new Uint32Array(cap*2)`, `set(old)`, swap. Amortized O(1); the copy is a fast memcpy with no GC tracing. Keep a scalar `length` — never rely on `typedArray.length` semantics for logical size.
- **ES2024 resizable ArrayBuffer** ([2ality](https://2ality.com/2024/06/array-buffers-es2024.html), [proposal](https://github.com/tc39/proposal-resizablearraybuffer)): `new ArrayBuffer(n, { maxByteLength })` grows **in place** (implementations reserve/commit virtual memory), and length-tracking TypedArrays auto-resize. Caveats straight from the proposal: TypedArrays on resizable buffers "are expected to be slower" — every index access carries a **generalized bounds check** (hoistable in tight loops like the detach check) and they get a **distinct hidden class** from fixed-length TypedArrays — so don't mix fixed and resizable-backed arrays at one access site (polymorphic IC). `SharedArrayBuffer` can only grow, never shrink, and isn't transferable. [`ArrayBuffer.prototype.transfer`](https://github.com/tc39/proposal-arraybuffer-transfer) gives move semantics (detach + reuse), often zero-copy for grow-in-place. As of 2024-2026 V8, sensible default: **fixed-length buffers + manual doubling** for hot arrays; resizable buffers are for convenience, not speed.

## 5. Prior art: reactivity + data-oriented / WASM

- **[Preact "Signal Boosting"](https://preactjs.com/blog/signal-boosting/)** — the definitive writeup of the Node-object linked-list design alien-signals descends from. Key extractable details: they moved **from `Set`s to doubly-linked Node objects** because Sets are expensive to construct per-recompute and iterate slightly slower than arrays; each Node is "a sort of quad-linked monstrosity" serving both the source list and target list (**one allocation per dependency-dependent pair, kept alive as long as the edge exists**); before each recompute, existing nodes are flagged unused and cached on `source.node`, reused if re-encountered, and only still-unused nodes are purged after; **version numbers** (per-signal + one global version) short-circuit recomputation; effect scheduling uses an intrusive singly-linked list (`.nextBatchedEffect` on the effect itself — zero allocation). Every one of these tricks has a direct SoA translation where the "allocation" becomes a free-list slot claim.
- **[reactively](https://github.com/milomg/reactively)** (milomg): plain JS **arrays** `sources[]`/`observers[]` instead of linked lists, with the index-pointer diffing described in §4, plus the clean/check/dirty (0/1/2) graph-coloring algorithm. Evidence that array-based adjacency is competitive with linked lists at the object level in JS — an SoA port removes its remaining allocation (array reallocations on divergence).
- **[leptos ARCHITECTURE.md](https://github.com/leptos-rs/leptos/blob/main/ARCHITECTURE.md)** (Rust): the entire reactive system is "a single data structure that exists at runtime"; **signals live in a slotmap arena (generational arena + free list), and the user-facing `Signal`/`Memo` types are just `Copy` indices into it**. This is the architectural blueprint for an index-handle signals API, proven in production. (Slotmaps also recommended for graph-heavy UI in [slint#5292](https://github.com/slint-ui/slint/issues/5292); see also ["Object Soup is Made of Indexes"](https://lobste.rs/s/zhbv0i/object_soup_is_made_indexes).)
- **No JS signals library on typed arrays found** despite targeted searching (GitHub repo/code searches for "signals typed array", "data-oriented reactivity", "signal graph arena", "reactivity arraybuffer" all came up empty). Vue's tracker only *supports observing* TypedArrays as values ([vuejs/core#802](https://github.com/vuejs/core/issues/802)). This is an open niche.
- **WASM reactivity — why the boundary kills it**: per-call JS↔WASM overhead is small (~100ns) but a signals API is *made of* tiny calls (`get`/`set` per dependency per recompute), which is exactly the death mode described in boundary-crossing analyses ([16 Patterns for Crossing the WebAssembly Boundary](https://dev.to/rafacalderon/16-patterns-for-crossing-the-webassembly-boundary-and-the-one-that-wants-to-kill-them-all-5kb)); "the more often you cross it and the more data you copy, the faster you kill all performance gains." Also, JS values can't cross without externref bookkeeping, so a WASM signals core could only store the *graph*, calling back into JS for every computed function — worst case for crossings. And Surma's [js-to-asc](https://surma.dev/things/js-to-asc/) numbers show even pure-WASM compute is no free lunch: blur benchmark JS/TurboFan **104.35ms** vs naive AssemblyScript **298.13ms**; only after `unchecked()` bounds-check removal (162.63ms) and `-O3s` (**89.60ms**) did ASC win; bubblesort ASC-optimized 65.42ms vs JS 103.85ms; but the allocation-heavy **binary-heap benchmark: JS 233.68ms vs AssemblyScript-with-GC 18,758ms (80x slower)**, fixed only by swapping to the no-GC `stub`/`minimal` runtimes (345/354ms, still 1.5x slower than JS). Lesson: V8's allocator+GC is *extremely* good; if you leave JS you must also bring a better allocator, and for a graph library the win, if any, comes from layout — which typed arrays give you *without* leaving JS. Leptos avoids all this only because the *entire app* lives in WASM.

## 6. GC pressure: what Link-object churn actually costs, and when pooling backfires

From [Trash talk: the Orinoco garbage collector](https://v8.dev/blog/trash-talk) and [Orinoco: young generation GC](https://v8.dev/blog/orinoco-parallel-scavenger):
- V8 is generational: young gen is a **semi-space Scavenger** — cost is **proportional to *surviving* objects, not allocations**. Dying-young garbage is nearly free. So the raw *allocation* of short-lived Link objects is cheaper than folk wisdom suggests; the real costs are (a) Links that *survive* scavenges get copied twice then **promoted to old space**, where they're touched by concurrent marking and possibly compaction; (b) every pointer store into a heap object runs a **write barrier**, and old→new pointers get recorded in a remembered set. A dependency graph that persists across frames means most Links are long-lived: they get promoted, and every re-link operation (splicing doubly-linked lists) is 4-6 barriered pointer stores.
- **Why pooling objects can backfire in JS**: pooled objects are by definition long-lived → promoted to old space; every time you "allocate" from the pool and wire a fresh young object (e.g. a new computed's node) into it, you create **old→new references that the write barrier must track and the scavenger must treat as roots**; pools also defeat the scavenger's "die young = free" economics and can extend marking time. Pooling wins only when objects are large/expensive to initialize or when churn survives scavenges anyway. (Anecdotal field report: [WebGL GC investigation](https://whenderson.dev/blog/webgl-garbage-collection/) — per-frame allocation caused multi-frame stalls and several-hundred-ms post-GC pauses; caching fixed it — externally-backed objects like textures/ArrayBuffers only die at *major* GC.)
- **The SoA move deletes the whole problem class**: indices in `Uint32Array`s are not pointers — no write barriers on edge splicing, nothing for the marker to trace, no promotion, no compaction moves invalidating anything. The only GC-visible objects left are the values side-array and user closures. This — not cache locality — is often the biggest practical win of typed-array graph layouts in JS (mraleph's §6 result and Surma's binary-heap disaster are both fundamentally allocator/GC stories).

## 7. Concrete design sketch distilled from all of the above

- **Node arena (SoA)**: `flags: Uint32Array`, `version: Uint32Array` (raw u32, no Smi cliff), `depsHead/depsTail/subsHead/subsTail: Uint32Array` (link indices, `0` or `0xFFFFFFFF` as NIL), plus `values: any[]` and `fn: Function[]` side arrays kept **packed** (grow by `push`, never `new Array(n)`, never delete — overwrite with `undefined`).
- **Link arena (SoA)**: `dep, sub, prevSub, nextSub, nextDep (+prevDep): Uint32Array` — 20-24B/edge vs ~40B+GC for a Link object; free list threaded through `nextDep` with a scalar `freeHead`; geometric doubling growth via `new Uint32Array(2n)` + `.set()`.
- **Handles**: either bitECS-style packed u32 with version bits for safety, or thin per-node wrapper objects (one shape for all node kinds → all ICs monomorphic; wrappers are per-node not per-edge so their cost is bounded).
- **Keep every hot access site monomorphic**: one function per operation touching one array kind; don't mix fixed-length and resizable-buffer-backed views at the same site; don't read out-of-bounds (use explicit NIL checks, never sentinel reads past `length`).
- Bit-pack where it helps: flags already are; consider packing `depth`/small metadata into spare flag bits rather than another array (one load instead of two).
- **Don't move the core to WASM**; if multithreading ever matters, the same SoA layout ports directly to `SharedArrayBuffer` + `Atomics` (bitECS advertises exactly this path).

## Caveats / open questions
- SMI plain arrays vs typed arrays for *read-heavy* workloads is close to a wash on modern V8 ([elements-kinds](https://v8.dev/blog/elements-kinds), [thlorenz v8-perf notes](https://github.com/thlorenz/v8-perf/blob/master/data-types.md)); the typed-array win is memory density, kind-stability, no write barriers, and immunity to holes/transitions — benchmark both for the link arena.
- The values side-array is `PACKED_ELEMENTS` (tagged pointers) regardless; a signals workload dominated by *value equality checks and user callbacks* rather than graph traversal will see smaller gains than source-map's 4.2x (which was parse/sort dominated).
- js-ecs-benchmarks-style ~2x iteration gains are for linear sweeps; signal graph traversal is pointer-chasing with poor locality either way — expect the win to come mostly from allocation/barrier elimination and IC monomorphism, and from *enabling* future locality tricks (e.g., allocating a node's links contiguously).
