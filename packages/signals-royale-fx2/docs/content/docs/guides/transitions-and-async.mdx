## Transition drafts

A write made inside `startSignalTransition` or `useSignalTransition` goes into
a draft. The committed tree keeps reading base state while the transition's
render passes include that draft.

```tsx
const [pending, start] = useSignalTransition()

start(() => count.update((n) => n + 10))
count.update((n) => n * 2) // urgent
```

<Playground name="transition" />

Updater functions replay in dispatch order. If the count begins at `1`, the
drafted `+10` happens before the urgent `×2` in the transition's snapshot, so
that future becomes `22`. The committed tree temporarily shows `2` because it
does not include the draft.

## Promises in computeds

The `use` argument passed to `createComputed` reads a stable thenable. A first
load suspends. A later load keeps serving the previous settled value while
`isPending` reports that newer data is behind it.

```tsx
const user = createComputed((use) => use(requestUser(userId.get())))

function User() {
	const value = useValue(user)
	const pending = useIsPending(user)
	return <p>{value.name}{pending ? ' (refreshing)' : ''}</p>
}
```

<Playground name="async" />

Keep thenables stable for one input set. If the computed creates a new promise
on every evaluation, each settlement starts another request and the signal can
never settle.

To refetch unchanged inputs, add a version atom to the request key and bump it.
That remains an ordinary write, so refetching composes with batching and
transitions without a separate lifecycle.
